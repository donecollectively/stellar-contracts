# Stellar Contracts v0.9.5-beta.1 Release Notes

*Released: February 2026*

---

## Application Performance

*UTxO Caching Layer* — dApps built on Stellar Contracts now have a persistent, browser-side UTxO store via `CachedUtxoIndex`. The result is a dApp that loads quickly, queries chain data without delays, and stays responsive even under heavy load or when sharing API capacity with other users. `CapoDappProvider` wires this in automatically.

**Startup Performance** — Several infrastructure enhancements to the framework's loading sequence result in noticeably faster dApp startup times.

---

## Reference Documentation and Agent Guidance

*Reference Documentation and Agent Guidance* — This release includes extensive additions to the reference library, with material organised for both human readers and AI-assisted development workflows. New and updated guides cover on-chain Helios patterns, off-chain transaction building, validity windows, byte comparison safety, full-stack architecture, and testing patterns — making it easier to develop, test, and maintain Stellar Contracts applications effectively.

*Time and validity window documentation* — Added cross-cutting validity window guidance across on-chain, off-chain, and testing reference docs. Covers the complete flow from building a time-bounded transaction to testing time-based rejections.

---

## Transaction-Building and On-Chain Data Lifecycle

*Activity introspection: module, name, and data* — The `isActivity` type now carries the module, activity name, and underlying data object — useful for `beforeCreate`/`beforeUpdate` hooks that need to vary behaviour by activity type.

*Transaction context in lifecycle hooks* — `beforeCreate()` and `beforeUpdate()` now receive the active `StellarTxnContext` (`tcx`). Useful for patterns like stamping a record's `createdAt` field with the transaction's network-aligned time.

*Faster on-chain data reading* — Data controllers initialized with `{ onchain: false }` skip Helios script compilation entirely, making record reads significantly cheaper. Scripts compile on demand only when a write transaction requires them.

---

## Testing Environment: On-Disk State Snapshots

*On-Disk State Snapshots for Testing* — Test suites no longer re-run expensive blockchain setup on every run. The emulator persists smart contract state, test-scenario actor wallets, and transaction history to disk, restoring in seconds. Only snapshots that have changed rebuild — the rest are reused as-is.

---

## Release Tooling and Repository Infrastructure

*Repository structure* — The repository now uses Weaver for multi-tree branch management, separating the library from tooling and documentation branches. No impact on library consumers.

*Structured logging (preview)* — A structured logging module (`stellog`) is in development. Not yet integrated into the framework; available for early experimentation.
