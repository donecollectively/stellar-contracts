# Detailed Release Notes — 0.9.5-beta.2

_Previous release: 0.9.5-beta.1_

**Download / package.json dependency:** <https://raw.githubusercontent.com/donecollectively/stellar-contracts/0.9.5-beta.2/stellar-contracts-0.9.5-beta.2.tgz>

### Multi-Tab Chain Synchronization (beta)

- Multi-tab chain synchronization (beta): Browser tabs sharing a Dexie UTxO store no longer risk data corruption. A sync mutex governs which tab syncs; a write lock protects all store mutations. Automatic stale-owner takeover ensures no tab is left without sync.

### Transaction Confidence and Recovery

- Confirmation depth tracking: Confirmed pending transactions progress through graduated confidence levels as blocks accumulate: provisional (depth < 4), likely (>=4), confident (>=10), certain (>=180 blocks). Thresholds are configurable via confirmationThresholds constructor parameter. updateConfirmationDepths() runs at each sync cycle end, advancing confirmState and emitting confirmStateChanged events at threshold crossings. getPendingTxs() accepts optional { confirmState } filter. Transactions removed from pendingTxHashes at 'likely' threshold rather than immediately on confirmation. The 'certain' threshold uses slot-based depth (1 hour) for timely feedback on pre-production networks.

- Background re-submission for reliability: The system silently resubmits pending and provisionally-confirmed transactions that have not reached sufficient depth, with eligibility guards and per-transaction throttling.

- Continuous chain synchronization: The system stays aligned with the canonical chain, automatically adjusting transaction status when the chain state evolves. Robust to normal micro-forks that can occur while chain state is settling.

- Pending transaction confirmation safety net: The sync cycle now includes a dedicated check for pending transactions that may have been confirmed while the browser was away. On reload, each pending transaction is verified against Blockfrost, and its status is updated accordingly.

- Graceful conflict resolution for pending transactions: When a confirmed transaction spends outputs that a pending transaction also claims, the system detects the conflict during block processing and defers cleanup. Cleanup proceeds only after the pending transaction's deadline has expired with sufficient block depth, and all conflicting transactions have reached sufficient confirmation depth. This prevents premature state changes during the period when micro-forks are still possible. The PendingTxTracker UI shows a rollback-pending indicator during this waiting period.

- Output verification on confirm: repairs partially-indexed transactions after interrupted registration. Metadata resolution with fallback paths. Depth-accurate confidence calculation.

- Separated chain tip tracking from block processing, fixing a bug where confirmed transactions remained in pending state. Two sync modes (incremental and catchup) with a concurrency guard.

- Prevent reversion of wallet UTxOs that belong to pending transactions when doing wallet reconciliation.

### Transaction Observability and Diagnostics

- Chain sync log viewer (beta): New IndexerLogViewer component for real-time visibility into chain synchronization operations. All sync logging migrated to structured helpers with log levels and parent-child grouping. Filterable, searchable, auto-tailing UI.

- Pending transaction indicators are now clickable, opening a detail panel with transcript, structure, diagnostics, and submission log tabs. Transaction diagnostic data is captured at signing and persisted for later inspection.

- Pending transactions record a structured event log covering resubmission, chain adjustments, and status changes, enabling post-hoc diagnosis of the transaction lifecycle.

- Transaction UI refinements: log-style submission rows, click-to-copy CBOR with floating toast, zoom/close controls, resubmit button on the detail panel, and selection indicators on pending transaction dots.

### Various Consistency Fixes

- Network time and wallet balance fixes: Transaction time, the indexer's clock, and wallet sync now use correct sources, eliminating time drift and stale balance display.

- Transaction submission resilience: Rate-limit errors, transient failures, and several state machine issues handled gracefully. Delegate setup recovers from partial failures automatically.

- Per-activity governance exemption: On-chain policies can now provide granular exemptions from the normal "governance authority required" rule for specific activities. Downstream delegates must add isGovExempt* functions.

### Release Infrastructure

- Release pipeline is now package-name agnostic. Output shows tarball URL and release notes links.

- scripts/release now supports --help with all bump modes computed from the current version, and validates unknown arguments.
