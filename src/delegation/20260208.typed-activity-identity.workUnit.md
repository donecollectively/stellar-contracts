## Problem / Context

`beforeCreate()` and `beforeUpdate()` hooks receive `activity: isActivity` in their context — an opaque UPLC redeemer that hooks cannot meaningfully inspect. Downstream controllers need to know WHICH activity variant triggered the hook and WHAT arguments were passed, so they can branch their record-fixup logic accordingly.

The generated type bridges already produce discriminated union types per controller:
```typescript
type MintingActivity = { CreatingTData: TxOutputId }
type SpendingActivity = { UpdatingTData: number[] }
```

This work unit adds type-safe `activityName` and `activityArgs` to both context types, using a distributing utility type that transforms the generated activity union into a discriminated union on `activityName`. TypeScript's native switch narrowing then gives exhaustiveness checking and implicit arg type narrowing in hook implementations.

Backwards-compatible: new generic type parameters default to `any`, so existing `DelegatedDataContract` subclasses that don't use the hooks need no changes.

## Plan / Guidance

### 1. Utility Types

Add to `src/helios/typeUtils.ts` (or a new utility file if the Coder prefers):

```typescript
/** Extract all variant names from a single-key-object union */
type ActivityVariantNames<MA> = MA extends any ? keyof MA : never;

/** Map variant name to its args type */
type ActivityVariantArgs<MA, K extends PropertyKey> =
    MA extends { [P in K]: infer V } ? V : never;

/** Transform { VariantName: ArgsType } union into discriminated union on activityName */
type ActivityIdentity<MA> = MA extends infer V
    ? { [K in keyof V]: { activityName: K; activityArgs: V[K] } }[keyof V]
    : never;
```

Verify with concrete types from `DelegatedDatumTester.typeInfo.d.ts` that switch narrowing works.

### 2. Generic Threading

Add optional type parameters with defaults to `DelegatedDataContract`:

```typescript
class DelegatedDataContract<
    T extends AnyDataTemplate<any, any>,
    TLike extends AnyDataTemplate<any, any>,
    MA = any,  // MintingActivity type from generated bridge
    SA = any   // SpendingActivity type from generated bridge
> extends ContractBasedDelegate
```

Update `createContext` and `updateContext`:

```typescript
type createContext<TLike, MA = any> = {
    activity: isActivity;
    tcx: StellarTxnContext;
} & ([MA] extends [any] ? { activityName?: string; activityArgs?: unknown } : ActivityIdentity<MA>);

type updateContext<T, SA = any> = {
    original: T;
    activity: isActivity;
    tcx: StellarTxnContext;
} & ([MA] extends [any] ? { activityName?: string; activityArgs?: unknown } : ActivityIdentity<SA>);
```

The conditional on `[MA] extends [any]` provides a graceful fallback: when the type parameter isn't specified, `activityName` is `string | undefined` and `activityArgs` is `unknown | undefined`. When specified, the full discriminated union applies.

**Important**: The hook method signatures on the class need to use the class-level `MA`/`SA` type parameters:
```typescript
beforeCreate(record: TLike, context: createContext<TLike, MA>): TLike
beforeUpdate(record: TLike, context: updateContext<T, SA>): TLike
```

### 3. Identity Extraction at Call Sites

Two sources of activity identity, depending on what's available:

**From `SeedActivity`/`UpdateActivity` wrappers** (preferred — pre-materialization):
- `SeedActivity` already holds the factory function and `arg`. Add a `variantName: string` field, set by the bridge-generated accessor (the variant name is known at code-generation time via the method name).
- Same pattern for `UpdateActivity`.

**From `isActivity.details` string** (fallback — post-materialization):
- The bridge generators already set `details` to `"ScriptName::ActivityType.VariantName"`. Parse with `details?.split('.').pop()`.
- This covers the case where a raw `isActivity` is passed directly (no wrapper).

### 4. SeedActivity / UpdateActivity Enhancement

Add `variantName` field to both classes:
```typescript
class SeedActivity<FactoryFunc extends seedActivityFunc<any, any>> {
    variantName: string;
    arg: SeedActivityArg<FactoryFunc>;
    constructor(host, factoryFunc, arg, variantName: string) { ... }
}
```

The bridge generators (specifically the `$seeded$` accessor pattern and `SpendingActivities`/`MintingActivities` nested accessors) already know the variant name — they include it in the `details` string. They need to also pass it as a parameter to `SeedActivity`/`UpdateActivity` constructors.

**Bridge generator impact**: The bridge generators are in `src/helios/scriptBundling/` — check if the variant name needs to be threaded through the template or if it can be extracted from existing data in the generation context. This is the most uncertain part of the scope.

## Advisory Notes

### Architect Notes (2026-02-08)

**Bridge generator uncertainty**: The plan calls for `SeedActivity`/`UpdateActivity` to carry `variantName` as a string, set during construction by the bridge-generated code. The bridge generators (`src/helios/scriptBundling/`) are template-driven. The Coder should investigate whether the variant name is readily available in the generation context or requires additional threading. If the bridge generator changes prove disproportionately complex, the `details` string parsing fallback is architecturally acceptable — it's set by the same generators and follows a reliable convention. The type-level discriminated union still works regardless of the runtime extraction method.

**Conditional type design**: The `[MA] extends [any]` conditional in the context types is the critical ergonomics decision. Verify that this correctly distinguishes "caller didn't specify MA" (default `any`) from "caller specified a concrete union type." If `[MA] extends [any]` doesn't behave as expected for all cases, consider an explicit sentinel type (e.g., `type UnspecifiedActivity = { __unspecified: true }`) as the default instead of `any`.

**Lazy redeemer interaction (ARCH-b4gv3t4jyj)**: When activity construction is deferred via `LazyRedeemerData`, the `activityName` should still be available (the variant is chosen at call time, only UPLC construction is deferred). The `activityArgs` may reflect partial args known at invocation time. The Coder should verify this doesn't create a type conflict.
