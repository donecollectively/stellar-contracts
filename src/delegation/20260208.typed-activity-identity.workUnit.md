## Problem / Context

`beforeCreate()` and `beforeUpdate()` hooks receive `activity: isActivity` in their context — an opaque UPLC redeemer that hooks cannot meaningfully inspect. Downstream controllers need to know WHICH activity variant triggered the hook and WHAT arguments were passed, so they can branch their record-fixup logic accordingly.

The generated type bridges already produce discriminated union types per controller:
```typescript
type MintingActivity = { CreatingTData: TxOutputId }
type SpendingActivity = { UpdatingTData: number[] }
```

This work unit adds type-safe `activityName` and `activityArgs` to both context types, using a distributing utility type that transforms the generated activity union into a discriminated union on `activityName`. TypeScript's native switch narrowing then gives exhaustiveness checking and implicit arg type narrowing in hook implementations.

Backwards-compatible: new generic type parameters default to `any`, so existing `DelegatedDataContract` subclasses that don't use the hooks need no changes.

## Plan / Guidance

### 1. Enrich `isActivity` with Semantic Identity

**Design center shift**: Rather than reconstructing activity identity at downstream call sites, enrich `isActivity` itself at its point of origin (`EnumBridge.mkUplcData`).

Update the `isActivity` type in `src/ActivityTypes.ts`:

```typescript
type isActivity = {
    redeemer: UplcData;
    details?: string;
    variantName: string;
    activityArgs?: unknown;
    isDeferredRedeemer?: boolean;
};
```

Update `EnumBridge.mkUplcData` in `src/helios/dataBridge/EnumBridge.ts` to populate identity at construction time. The `enumPathExpr` parameter already contains the qualified variant name (e.g., `"DelegateDatumTester::SpendingActivity.UpdatingTData"`). Extract the short variant name from this, and pass the pre-serialization args through:

```typescript
if (this.isActivity) {
    return {
        redeemer: uplc,
        details: enumPathExpr,
        variantName: enumPathExpr.split('.').pop()!,
        activityArgs: /* pre-serialization value passed to the bridge method */,
    } as uplcReturnType;
}
```

**Choke point**: `EnumBridge.mkUplcData` is the single place where all `isActivity` objects are constructed. Populating identity here means no downstream extraction logic, no wrapper inspection, no string parsing at call sites.

**`activityArgs` threading**: The bridge-generated variant methods (e.g., `CreatingTData(seed, ...)`, `UpdatingTData(id)`) receive the pre-serialization args. These need to be captured and passed through to `mkUplcData`. Investigate how the `value` parameter to `mkUplcData` relates to the original args — it may already be them, or it may be the serialized form. The Coder should check this.

**`isDeferredRedeemer`**: Set to `true` when the activity wraps a `LazyRedeemerData`. At hook invocation time with lazy redeemers, the variant name is still known (chosen at call time), and `activityArgs` reflects partial args known at construction time.

### 2. SeedActivity / UpdateActivity: Thread Identity Through Wrappers

`SeedActivity` and `UpdateActivity` hold pre-materialization state. They call factory functions that ultimately produce `isActivity` via `mkUplcData`. After step 1, the materialized `isActivity` already carries identity.

However, before materialization (e.g., in `mkTxnCreateRecord` where `effectiveActivity` is still a `SeedActivity`), the wrapper should also expose identity for framework code that needs it pre-materialization:

```typescript
class SeedActivity<FactoryFunc extends seedActivityFunc<any, any>> {
    variantName: string;
    arg: SeedActivityArg<FactoryFunc>;
    constructor(host, factoryFunc, arg, variantName: string) { ... }
}
```

Same for `UpdateActivity`. The bridge-generated `$seeded$` accessors and `SpendingActivities`/`MintingActivities` nested accessors know the variant name at code-generation time — pass it to the constructor.

### 3. Utility Types

Add to `src/helios/typeUtils.ts` (or a new utility file if the Coder prefers):

```typescript
/** Transform { VariantName: ArgsType } union into discriminated union on activityName */
type ActivityIdentity<MA> = MA extends infer V
    ? { [K in keyof V]: { activityName: K; activityArgs: V[K] } }[keyof V]
    : never;
```

Verify with concrete types from `DelegatedDatumTester.typeInfo.d.ts` that switch narrowing works.

### 4. Generic Threading

Add optional type parameters with defaults to `DelegatedDataContract`:

```typescript
class DelegatedDataContract<
    T extends AnyDataTemplate<any, any>,
    TLike extends AnyDataTemplate<any, any>,
    MA = any,  // MintingActivity type from generated bridge
    SA = any   // SpendingActivity type from generated bridge
> extends ContractBasedDelegate
```

Update `createContext` and `updateContext`:

```typescript
type createContext<TLike, MA = any> = {
    activity: isActivity;
    tcx: StellarTxnContext;
} & ([MA] extends [any] ? { activityName?: string; activityArgs?: unknown } : ActivityIdentity<MA>);

type updateContext<T, SA = any> = {
    original: T;
    activity: isActivity;
    tcx: StellarTxnContext;
} & ([SA] extends [any] ? { activityName?: string; activityArgs?: unknown } : ActivityIdentity<SA>);
```

The conditional provides a graceful fallback: when the type parameter isn't specified (default `any`), `activityName` is `string | undefined` and `activityArgs` is `unknown | undefined`. When specified, the full discriminated union applies.

**Important**: The hook method signatures on the class need to use the class-level `MA`/`SA` type parameters:
```typescript
beforeCreate(record: TLike, context: createContext<TLike, MA>): TLike
beforeUpdate(record: TLike, context: updateContext<T, SA>): TLike
```

### 5. Call Site Changes

After step 1, `isActivity` objects already carry `variantName` and `activityArgs`. The hook call sites in `txnCreatingRecord` (line ~519) and `txnUpdatingRecord` (line ~671) pass the `activity` object in the context — the identity fields are already present on it.

The context types expose these via the intersection with `ActivityIdentity<MA>`/`ActivityIdentity<SA>`. The Coder needs to verify that the runtime values on the `isActivity` object satisfy the type-level discriminated union. This may require a well-placed type assertion at the framework boundary where the context is constructed — one assertion per call site, justified by the bridge generator contract.

**Core options types**: Add `activityName`/`activityArgs` to `CoreDgDataCreationOptions` and `CoreDgDataUpdateOptions` to thread identity from the public methods to the internal hook call sites. Do NOT add these to the public `DgDataCreationOptions` or `DgDataUpdateOptions` — the framework extracts identity from the activity, callers don't provide it.

## Advisory Notes

### Key Design Revision (Stakeholder-Directed, 2026-02-08)

The original plan had downstream code reconstructing activity identity from `SeedActivity`/`UpdateActivity` wrappers and `isActivity.details` string parsing — scattered extraction at multiple call sites. During Code Whisperer pre-coding review, finding **jp3zpynehb** revealed that `isActivity.details` is never actually populated by `EnumBridge.mkUplcData`, breaking the planned fallback.

The stakeholder directed a design center shift: **enrich `isActivity` itself** at its point of origin with `variantName`, `activityArgs`, and `isDeferredRedeemer`. This makes `isActivity` a self-describing activity envelope rather than an opaque UPLC wrapper. `EnumBridge.mkUplcData` — the single construction choke point for all activity objects — populates identity at birth. No downstream extraction logic needed.

This resolved both Code Whisperer Concerns (jp3zpynehb and spgmtnfvgg) and produced a cleaner architecture: identity lives where it originates, not reconstructed where it's consumed.

### Architect Notes (2026-02-08)

**Design center**: `isActivity` becomes a self-describing activity envelope. Identity lives where it originates (`EnumBridge.mkUplcData`), not reconstructed downstream. This is a conceptual shift — update JSDoc to reflect the expanded contract.

**`activityArgs` threading through EnumBridge**: The bridge-generated variant methods receive pre-serialization args, then call `mkUplcData(value, enumPathExpr)`. The `value` parameter is the serialization-ready form. Investigate whether `value` can serve directly as `activityArgs` or whether the original args need separate threading. This is the most uncertain part of the scope.

**Conditional type design**: The `[MA] extends [any]` / `[SA] extends [any]` conditionals in the context types are the critical ergonomics decision. Verify that these correctly distinguish "caller didn't specify" (default `any`) from "caller specified a concrete union type." If they don't behave as expected, consider an explicit sentinel type (e.g., `type UnspecifiedActivity = { __unspecified: true }`) as the default instead of `any`.

**Lazy redeemer interaction (ARCH-b4gv3t4jyj)**: When activity construction is deferred via `LazyRedeemerData`, `activityName` is still available (variant chosen at call time). `activityArgs` reflects partial args known at construction time. `isDeferredRedeemer` signals the deferred state.
