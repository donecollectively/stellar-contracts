## Problem / Context

`hasNamedSnapshot` decorator accepts an `actor` field specifying which actor should be active for that snapshot. When a snapshot is loaded from cache, `loadCachedSnapshot` correctly calls `setActor(actorName)` (line ~1086). However, when a child snapshot needs to be **built** (cache miss) while its parent is **cached** (cache hit), `ensureSnapshotCached` loads the parent's network state and named records but does NOT restore the actor.

**Observed behavior**: The content builder runs with whatever actor happened to be active before the snapshot system was invoked — which can be any actor (observed: "tina" when "marissa" was expected). If the builder doesn't defensively call `setActor()`, it operates as the wrong actor, causing failures like "no member token found" when the wrong actor's wallet is searched.

**Expected behavior**: The content builder should run with the decorator's specified actor already set, consistent with how `loadCachedSnapshot` handles the cached-load path.

**Root cause**: The SnapWrap contentBuilder lambda (line ~563) calls `generateSnapshotFunc` without first setting the declared actor. The post-build assertion (b) verifies the actor but no pre-build setup (a) ensures it. When the parent is loaded from cache, the actor is stale from a prior operation.

**Reproduction** (confirmed scientifically):
1. Build full snapshot chain from clean cache → all builders run, each calling `setActor()` internally → works
2. Run again (all cached) → `loadCachedSnapshot` sets actor from decorator → works
3. Delete only the child snapshot, keep parent cached → parent loaded from cache (no `setActor`), child builder runs with wrong actor → **fails**

## Focus Files

## Plan / Guidance

### 1. Add pre-build actor setup in contentBuilder lambda

In `CapoTestHelper.ts`, the SnapWrap contentBuilder lambda (line ~563), add actor setup before calling `generateSnapshotFunc`:

```typescript
() => {
    // (a) Pre-build: set declared actor (skip genesis — actors don't exist yet)
    const actorReady = parentSnapName !== "genesis"
        ? (actorName === "default"
            ? this.setDefaultActor()
            : this.setActor(actorName))
        : Promise.resolve();

    return actorReady.then(() =>
        generateSnapshotFunc.apply(this, args)
    ).then((result: any) => {
        // (b) Post-build: existing assertion (unchanged)
        ...
    });
},
```

### 2. No changes to `ensureSnapshotCached` or `findOrCreateSnapshot`

No signature changes. No parameter threading. The caching chokepoint is unaffected.

### 3. Verify no regressions

Builders with explicit `setActor()` calls override the pre-build default. The post-build assertion catches drift.

## Advisory Notes

**Minimal change**: This is a small, well-scoped fix. The actor-setting logic already exists in `loadCachedSnapshot` — the fix duplicates it to the build path.

**Downstream impact**: Once fixed, snapshot builders in consumer projects will no longer need defensive `setActor()` calls. Existing defensive calls become harmless redundancy and can be removed at the consumer's discretion.

**Discovery context**: Found while investigating "no member token found" failure in form-manager's `firstDriverChange` snapshot builder. The builder's parent (`firstAdoptedDriver`) was cached, but `firstDriverChange` wasn't. The actor defaulted to "tina" (arbitrary) instead of the decorator's "marissa", causing member token lookup to search the wrong wallet.

## Audit

### Architect

**Diagnosis**: The work unit correctly identifies the missing behavior — the decorator's `actor` field should be set before building (a), asserted after building (b), and set after loading (c). Currently (b) and (c) exist but (a) is missing. The reproduction scenario and root cause analysis are sound.

**Fix location correction**: The proposed fix threads `actorName` through `ensureSnapshotCached`, but `ensureSnapshotCached` is the **caching chokepoint** (ARCH-d52nrfd96z). Its responsibility is cache resolution, not actor lifecycle. The fix belongs in the **contentBuilder lambda** inside SnapWrap (`CapoTestHelper.ts:563-582`), which already owns (b) the post-build assertion and has `actorName` and `parentSnapName` in its closure.

**Design clarification**: The `actor` field in `@hasNamedSnapshot` is declarative — "this snapshot's world is this actor." The infrastructure honors it in all paths:
- **(a) Pre-build**: contentBuilder lambda sets actor before calling `generateSnapshotFunc` (MISSING — this is the fix)
- **(b) Post-build**: contentBuilder lambda asserts actor wasn't changed (EXISTS — lines 568-578)
- **(c) Post-load**: `loadCachedSnapshot` sets actor for the test (EXISTS — lines 1082-1087)

### Code Whisperer

**Structural concern (resolved)**: The work unit's original plan would spread actor-setting logic across three locations — `ensureSnapshotCached` (pre-build), SnapWrap lambda (post-build assertion), and `loadCachedSnapshot` (post-load). This is the "Scattered Heuristics" indicator: same concern, three touch points.

**Recommended placement**: Co-locate (a) with (b) in the contentBuilder lambda. This gives two locations, each owning one path cleanly:
- **Build path**: contentBuilder lambda owns (a) pre-build setup + (b) post-build assertion
- **Load path**: `loadCachedSnapshot` owns (c) post-load actor setting

**No signature changes needed**: The lambda already closes over `actorName` and `parentSnapName`. No threading through `ensureSnapshotCached` required. The caching chokepoint stays focused on caching.

### Domain-Fit Tester (stellar-testing)

**Genesis edge case**: For root snapshots (`parentSnapName === "genesis"`), actors don't exist yet — the builder CREATES them. Pre-build actor setting must be conditional: skip when `parentSnapName === "genesis"`. The builder handles its own actor setup in this case; the post-build assertion (b) still verifies.

**Recursive parent builds**: When `ensureSnapshotCached` recursively builds a parent via `parentReg.snapMethod.call(this)` (line 1146), the parent's SnapWrap runs the full `findOrCreateSnapshot` → `loadCachedSnapshot` path, which sets the PARENT's actor. Then `ensureSnapshotCached` reloads parent state from cache (line 1165) without touching the actor. So the child's contentBuilder starts with the parent's actor active, then the pre-build setup (a) changes it to the child's actor. Correct sequence — no conflict.

**Override scenario**: Builders that explicitly call `setActor()` mid-build (e.g., switching to a different actor for a specific transaction) will override the pre-build setup. The post-build assertion (b) then verifies the builder left the DECLARED actor active at the end, catching accidental actor drift.
