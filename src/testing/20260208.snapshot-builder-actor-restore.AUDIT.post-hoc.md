On cache miss, the actor is set in contentBuilder (pre-build), then `loadCachedSnapshot` sets it again (post-load). This is correct by design:

2. `loadCachedSnapshot` provides uniform loading for both hit and freshly-built paths

The double-set is harmless — `setActor` is idempotent for the same actor name.

## Stellar Testing: Post-Fix Review

### Scope

This work unit modifies testing infrastructure (`CapoTestHelper.ts`), not test code. The review assesses the fix from the testing domain perspective: correctness across snapshot cache paths, edge case coverage, and downstream impact on test suites.

### Scenario Analysis

The bug manifests only in the **partial-cache scenario** (parent cached, child uncached). Three cache scenarios exist:

| Scenario | Parent | Child | Actor Set By | Pre-Fix | Post-Fix |
|----------|--------|-------|--------------|---------|----------|
| Clean build | uncached | uncached | Parent builder leaves actor → child builder inherits | Works (incidentally) | Works (explicitly) |
| Full cache | cached | cached | `loadCachedSnapshot` (c) | Works | Works (unchanged) |
| Partial cache | cached | uncached | **Nothing** (pre-fix) / **contentBuilder (a)** (post-fix) | **FAILS** — stale actor | Works |

The fix closes the gap in scenario 3. The post-build assertion (b) would have caught this as a failure after the fact, but the pre-build setup (a) prevents the wrong-actor execution from happening at all — correct placement.

### Genesis Edge Case Verification

The genesis guard (`parentSnapName !== "genesis"`) is correct:
- `bootstrapWithActors` has `parentSnapName: "genesis"` and `actor: "default"` — its builder creates actors via `setupActors()`, so pre-build actor setup would fail (no actors exist yet)
- The post-build assertion (b) still runs for genesis, verifying the builder set a default actor by the end
- `loadCachedSnapshot` restores actors from stored keys (lines 1067-1077) before the actor-set call (line 1092-1097), so the load path handles genesis correctly

### Recursive Build Verification

When `ensureSnapshotCached` recursively builds a parent (line 1156: `parentReg.snapMethod.call(this)`), the parent's full SnapWrap executes, including its own pre-build actor setup. Then `ensureSnapshotCached` reloads parent state (line 1175) without touching the actor. The child's contentBuilder then sets its own actor via (a). Sequence is correct — no actor leak between parent and child builds.

### Downstream Test Impact

**Existing tests**: No test code changes needed. The fix makes the infrastructure honor the `actor` declaration consistently. Tests that already worked continue to work. Tests that relied on incidental actor ordering (clean-build scenario) now have explicit actor setup — more deterministic, not less.

**Defensive `setActor()` calls**: Downstream builders that defensively called `setActor()` at the start of their build functions (e.g., `CapoWithoutSettingsTestHelper.firstReqt()` per the work unit) now have redundant calls. These are harmless — `setActor()` is idempotent. They can be removed at consumer discretion but don't need to be.

**Override scenario**: Builders that switch actors mid-build (e.g., setting up a transaction as actor A, then switching to actor B for another) will override the pre-build setup. The post-build assertion (b) catches any actor left in the wrong state at build end. This is the existing contract — unchanged by the fix.

### Findings

#### C-1: Missing regression test for partial-cache actor setup (Concern — medium priority)

**Raised by**: Human tester, endorsed by Stellar Testing + Code Whisperer

**Evidence**: The bug only manifests in the partial-cache scenario (parent cached on disk, child uncached). No automated test reproduces this specific cache state. The coverage assessment initially called this "adequate" because every snapshot build exercises the contentBuilder lambda — but that's incidental coverage, not a regression guard. If someone removes the pre-build actor setup (a), no test goes red.

The work unit's reproduction steps (delete child cache, keep parent) are a manual diagnostic procedure, not an automated regression test.

**Dialogue**:
- *Human tester*: The coder didn't add a test to prevent regression or demonstrate with a red/green cycle the problem actually occurring.
- *Stellar Testing*: Valid — "exercised by every snapshot build" is not the same as a test that specifically guards the partial-cache scenario.
- *Code Whisperer*: "IMPLEMENTED/NEEDS VERIFICATION" without an automated verification path is not real verification.
- *Human tester*: The test should load the deep snap, explicitly delete the deepest cache file only, clear the memory cache, then load the deep snap again.

**Proposed test approach** (human tester's design):

1. Load a deep snapshot (full chain warm — memory + disk)
2. Delete only the deepest snapshot's cache file on disk
3. Clear the in-memory snapshot cache
4. Load the deep snapshot again

This reproduces the exact production scenario: parent hits disk cache (sets parent's actor), child misses (triggers build). Pre-fix, the child builder runs with the parent's stale actor. Post-fix, the pre-build setup (a) sets the child's declared actor before the builder executes.

**Why this approach is superior to in-memory simulation**: It exercises the real code path end-to-end — `ensureSnapshotCached` loading parent from disk, then invoking the child's contentBuilder. No synthetic setup or mocking of cache state. The test would have been red before the fix and green after — a true red/green regression cycle.

**Verification point**: The child's builder must observe the correct actor *during* execution, not just after. The simplest assertion is that the builder succeeds without a defensive `setActor()` call — if the wrong actor is active, it will fail to find the expected member token (the original bug symptom). Alternatively, the builder can record `this.actorName` at entry and assert it matches the declared actor.

**Action**: New work unit needed to implement this regression test. Medium priority — the fix is correct and the post-build assertion (b) provides a safety net, but the partial-cache path lacks automated verification.
