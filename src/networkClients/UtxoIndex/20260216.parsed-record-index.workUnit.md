## Problem / Context

CachedUtxoIndex stores raw inline datum CBOR but provides no way to access parsed, typed delegated-data records. Consumers must either call `Capo.findDelegatedDataUtxos()` (which re-fetches UTXOs from network and re-parses every datum each time) or manually decode CBOR themselves.

The indexer should be able to store pre-parsed records so that lookups by record-id or record-type are instant local reads against IndexedDB, with no network calls or CBOR decoding at query time.

A complication: in the CapoDappProvider lifecycle, the CachedUtxoIndex is often created **before** the Capo exists (using precompiled bundle config). The Capo is needed for datum parsing (it provides `getDgDataController()` and `newReadDatum()`). This means the index must support attaching the Capo later and catching up on any UTXOs that were indexed in raw-only mode.

### Prerequisite gap

REQT/6h4f158gvs specifies `blockHeight` on the utxos Dexie schema, but the implementation is missing it — neither `UtxoIndexEntry` nor the Dexie schema include `blockHeight`. This must be fixed first, as the catchup query (REQT/3aew7g7wdw) depends on querying UTXOs by block height.

## Plan / Guidance

### Phase 0: Prerequisite — Restore `blockHeight` to UTXO entries

1. Add `blockHeight: number` to `UtxoIndexEntry` type
2. Add `blockHeight` to `dexieUtxoDetails` entity
3. Update Dexie schema: `utxos: "utxoId, *uutIds, address, blockHeight"`
4. Populate `blockHeight` in `txOutputToIndexEntry()`, `txInputToIndexEntry()`, and `blockfrostUtxoToIndexEntry()` — requires passing block height through the call chain
5. Bump Dexie schema version

### Phase 1: Storage layer — RecordIndexEntry type and records table

1. Create `types/RecordIndexEntry.ts` — Helios-free type with `id`, `utxoId`, `type`, `parsedData` (JSON string), `spentInTx`
2. Add record methods to `UtxoStoreGeneric`: `saveRecord()`, `findRecord(id)`, `findRecordsByType(type)`, `markRecordSpent(id, txHash)`
3. Add metadata methods: `getLastParsedBlockHeight()`, `setLastParsedBlockHeight(height)`
4. Implement in `DexieUtxoStore`: `records` table with indexes `id` (PK), `utxoId`, `type`; metadata storage for `lastParsedBlockHeight`
5. Wire `markRecordSpent()` into existing `markUtxoSpent()` flow

### Phase 2: Refactor datum parsing out of Capo.findDelegatedDataUtxos()

The inline lambda in `findDelegatedDataUtxos()` (lines ~3488–3573) contains two reusable pieces:

1. **Datum type extraction** — peeks into UplcData to read the `tpe` field from the datum map. Pure CBOR/UplcData inspection, no Capo needed. Extract into a standalone function (e.g. `extractDelegatedDatumType(uplcData): string | undefined`).

2. **Controller-based parsing** — calls `capo.getDgDataController(type, { onchain: false })` then `controller.newReadDatum(datum.data)` to get `data.capoStoredData.data`. Extract into a helper (e.g. `parseDelegatedDatum(capo, datumType, uplcData): { id, type, ...fields } | undefined`).

After refactoring, `findDelegatedDataUtxos()` should call these extracted functions instead of inlining the logic. This keeps backward compatibility while enabling CachedUtxoIndex to reuse the same parsing.

### Phase 3: CachedUtxoIndex integration

1. **`attachCapo(capo)`** — stores the Capo reference, triggers catchup
2. **Inline parsing during monitoring** — in `processTransactionForNewUtxos()`, after `indexUtxoFromOutput()`, if Capo is attached: decode `inlineDatum` CBOR → extract type → parse via controller → `store.saveRecord()`. Uses the refactored helpers from Phase 2.
3. **Catchup** — query UTXOs where `blockHeight > lastParsedBlockHeight`, `inlineDatum != null`, `spentInTx == null`. Parse each, save records, advance `lastParsedBlockHeight`.

### Phase 4: CapoDappProvider integration

In `connectCapo()`, after the Capo is created and the utxoIndex is available, call `utxoIndex.attachCapo(capo)`. This is a single line addition in both Path 1 (early index) and Path 2 (late index) flows.
