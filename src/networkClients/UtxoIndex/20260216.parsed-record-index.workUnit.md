## Problem / Context

CachedUtxoIndex stores raw inline datum CBOR but provides no way to access parsed, typed delegated-data records. Consumers must either call `Capo.findDelegatedDataUtxos()` (which re-fetches UTXOs from network and re-parses every datum each time) or manually decode CBOR themselves.

The indexer should be able to store pre-parsed records so that lookups by record-id or record-type are instant local reads against IndexedDB, with no network calls or CBOR decoding at query time.

A complication: in the CapoDappProvider lifecycle, the CachedUtxoIndex is often created **before** the Capo exists (using precompiled bundle config). The Capo is needed for datum parsing (it provides `getDgDataController()` and `newReadDatum()`). This means the index must support attaching the Capo later and catching up on any UTXOs that were indexed in raw-only mode.

### Prerequisite gap

REQT/6h4f158gvs specifies `blockHeight` on the utxos Dexie schema, but the implementation is missing it — neither `UtxoIndexEntry` nor the Dexie schema include `blockHeight`. This must be fixed first, as the catchup query (REQT/3aew7g7wdw) depends on querying UTXOs by block height.

## Plan / Guidance

### Phase 0: Prerequisite — Restore `blockHeight` to UTXO entries

1. Add `blockHeight: number` to `UtxoIndexEntry` type
2. Add `blockHeight` to `dexieUtxoDetails` entity
3. Update Dexie schema: `utxos: "utxoId, *uutIds, address, blockHeight"`
4. Populate `blockHeight` in `txOutputToIndexEntry()`, `txInputToIndexEntry()`, and `blockfrostUtxoToIndexEntry()` — requires passing block height through the call chain
5. Bump Dexie schema version

### Phase 1: Storage layer — RecordIndexEntry type and records table

1. Create `types/RecordIndexEntry.ts` — Helios-free type with `id`, `utxoId`, `type`, `parsedData` (JSON string), `spentInTx`
2. Add record methods to `UtxoStoreGeneric`: `saveRecord()`, `findRecord(id)`, `findRecordsByType(type)`, `markRecordSpent(id, txHash)`
3. Add metadata methods: `getLastParsedBlockHeight()`, `setLastParsedBlockHeight(height)`
4. Implement in `DexieUtxoStore`: `records` table with indexes `id` (PK), `utxoId`, `type`; metadata storage for `lastParsedBlockHeight`
5. Wire `markRecordSpent()` as an internal cascade from `DexieUtxoStore.markUtxoSpent()` — the store owns spent-state consistency. Cascade MUST look up by `utxoId` (not record `id`) to avoid marking a replacement record as spent. With `id` as PK and `put()` semantics, a record update within the same transaction is safe: output processing saves the new record (overwriting the old by `id`), then input processing cascades on the old `utxoId` which is no longer in the records table (no-op). **Processing-order invariant**: `processTransactionForNewUtxos()` MUST process outputs before inputs to maintain this guarantee.

### Phase 2: Extract `parseDelegatedDatum()` method on Capo

The inline lambda in `findDelegatedDataUtxos()` (lines ~3488–3581) encapsulates three concerns that should be a single reusable method on the Capo class:

**New method: `async parseDelegatedDatum(uplcData: UplcData, charterData?: CharterData): Promise<{ id: string; type: string; data: Record<string, any> } | undefined>`**

Encapsulates the full datum → typed record pipeline:
1. **Datum type extraction** — inspect `UplcData.fields[0]` map for the `tpe` key to get `datumType: string`. Return `undefined` for datums with no type field (e.g. ScriptReference, non-delegated-data).
2. **Controller lookup** — `this.getDgDataController(datumType, { charterData, optional: true, onchain: false })`. Return `undefined` if no controller registered for this type.
3. **Parse + unwrap** — `controller.newReadDatum(uplcData)` → extract `.capoStoredData.data` → return `{ id, type: datumType, data: typedData }`.

After extraction, refactor `findDelegatedDataUtxos()` to call `this.parseDelegatedDatum(datum.data, charterData)` instead of inlining the logic. This eliminates the `as any` cast and makes the parsing pipeline available to CachedUtxoIndex.

**No cross-module extraction** — all datum structure knowledge stays in Capo. CachedUtxoIndex consumes the method via the attached Capo instance.

### Phase 3: CachedUtxoIndex integration

1. **`attachCapo(capo)`** — accepts a Capo instance via `import type { Capo }` (type-only, no runtime coupling). Stores the reference and triggers catchup. CachedUtxoIndex gains no new runtime dependency on Capo — the caller (CapoDappProvider) already has the instance and passes it through.
2. **Inline parsing during monitoring** — in `processTransactionForNewUtxos()`, after `indexUtxoFromOutput()`, if Capo is attached: decode `inlineDatum` CBOR hex to `UplcData` via `decodeUplcData(hexToBytes(entry.inlineDatum))`, then call `capo.parseDelegatedDatum(uplcData)`. If result is non-undefined, call `store.saveRecord()` with the `{ id, type, data }` result.
3. **Catchup** — query UTXOs where `blockHeight > lastParsedBlockHeight`, `inlineDatum != null`, `spentInTx == null`. For each, decode CBOR and call `capo.parseDelegatedDatum()`. Save records, advance `lastParsedBlockHeight`.

### Phase 4: CapoDappProvider integration

In `connectCapo()`, after the Capo is created and the utxoIndex is available, call `utxoIndex.attachCapo(capo)`. This is a single line addition in both Path 1 (early index) and Path 2 (late index) flows.

### Pre-Work Collaborative Review (2026-02-16)

#### Architect leads — findings

##### Finding F1: Capo coupling approach unspecified

Phase 2 extracts datum-parsing helpers from `Capo.findDelegatedDataUtxos()` and Phase 3 introduces `attachCapo(capo)` on CachedUtxoIndex. The architecture doc explicitly documents CachedUtxoIndex as **decoupled from Capo** — it accepts discrete components (address, mph, bridge) to avoid circular dependencies. The work unit doesn't address:
- Where the extracted helpers live (dependency direction)
- Whether `attachCapo()` should accept a Capo type directly or an interface/protocol
- How the documented decoupled architecture should be updated

> **Resolution**: `attachCapo()` accepts a Capo instance via `import type` — type-only, no runtime coupling. The type boundary is preserved (same pattern as existing `CharterData` type import). All datum-parsing knowledge stays in Capo as a new `parseDelegatedDatum()` method that encapsulates type extraction, controller lookup, and parsing. Phase 2 becomes an extract-method refactoring within Capo only — no cross-module helpers. `findDelegatedDataUtxos()` calls it internally; CachedUtxoIndex calls it via the attached Capo instance. Architecture doc to be updated with optional datum-parsing attachment. Unit testing scoped out. (commit: see below)

#### Code Whisperer — sequenced follow-up

##### Finding CW-F1: markUtxoSpent / markRecordSpent coupling risk

In `processTransactionForNewUtxos()` (line ~636), spent UTXOs are marked via `store.markUtxoSpent()`. Phase 1 step 5 says "Wire `markRecordSpent()` into existing `markUtxoSpent()` flow." If this means adding a separate `store.markRecordSpent()` call alongside `store.markUtxoSpent()` in the orchestrator, these become paired operations that must always be called together — if one is missed, UTXO/record spent state becomes inconsistent. The store should own this consistency: `markUtxoSpent()` should internally cascade to `markRecordSpent()` for any corresponding record.

> **Arch**: Cascade must look up by `utxoId`, not record `id` — this prevents marking a replacement record as spent when old and new appear in the same transaction. Processing-order invariant (outputs before inputs) plus `id`-as-PK with `put()` overwrites ensures no availability gap during record updates. Catchup's `spentInTx == null` filter naturally converges to the latest version.

> **Resolution**: Phase 1 step 5 updated to specify internal cascade in `DexieUtxoStore.markUtxoSpent()`, cascade by `utxoId`, and the processing-order invariant. (commit: see below)
