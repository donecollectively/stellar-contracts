## Problem / Context

The `PendingTxTracker` shows compact dots for in-flight transactions but provides no drill-down capability ‚Äî clicking a dot does nothing. Users can only see details during the active batch session via `TxBatchViewer`, but once the batch is rotated or the page is reloaded, all diagnostic information (build transcript, tx structure, submitter state) is lost.

Three connected problems:

1. **No persistent diagnostics**: The submission pipeline (`TxSubmitMgr`) tracks rich state (submit attempts, confirmations, backoff, errors) but only in memory. None of this survives page reload.

2. **Tracker lifecycle tied to batch**: `TxSubmissionTracker` objects are owned by `BatchSubmitController` and destroyed when the batch rotates. But the transactions they manage are still in-flight ‚Äî the trackers should outlive the batch.

3. **No detail-on-demand from PendingTxTracker**: The dots have no onClick behavior. The existing `ShowTxDescription` component in `TxBatchViewer` has transcript/structure/diagnostics tabs, but it's private and requires live objects.

### Design

The solution has four parts:

**A. Persistent diagnostic fields on PendingTxEntry**: Capture `buildTranscript` (logger history), `txStructure` (dumpAny output), `signedTxStructure`, and `submissionLog` (incremental event log) at registration/signing time. These survive in Dexie across page reloads.

**B. Submission log written incrementally**: `TxSubmitMgr` writes `SubmissionLogEntry` events on each state transition via a callback. Events are appended to the Dexie `PendingTxEntry.submissionLog` via `appendSubmissionLog()`.

**C. Tracker lifecycle transfer to TxBatcher**: When a `TxSubmissionTracker`'s tx is signed and submission begins, the tracker is registered with `TxBatcher` by txHash. `TxBatcher` owns its lifecycle from that point ‚Äî the tracker survives batch rotation and is destroyed only at terminal state (confirmed-certain or rolled-back).

**D. Shared TxDetailPanel with hybrid data sourcing**: Refactor `ShowTxDescription` ‚Üí exported `TxDetailPanel`. `PendingTxTracker` onClick opens it, looking up a live tracker via `txBatcher.findTracker(txHash)` when available, falling back to Dexie data when not.

### Dropped: `signedTxStructure` field

The `signedTxStructure` field was dropped from scope. The signed CBOR (`signedTxCborHex`) is already persisted in `PendingTxEntry` and can be decoded on demand. The unsigned `txStructure` captures the meaningful structural content (inputs, outputs, mints); the signed version only adds the txHash (which changes after signing) and witness data. ### Design Decision: TxSubmitMgr Submission Log Emission (Topic 2.1)

**Intent**: After page reload, the user can see what happened during submission ‚Äî which submitters tried, what failed/succeeded, retries, errors ‚Äî as a timestamped event log persisted in Dexie.

**Options evaluated**:

**Option A: Explicit callback on TxSubmitMgr constructor** ‚Äî Add optional `onSubmissionLog` callback. TxSubmitMgr calls it at ~8-10 specific code points (`trySubmit()` ‚Üí `"submit-attempt"`, `notSubmitted()` ‚Üí `"submit-failed"`, `didConfirm()` ‚Üí `"confirmed"`, etc.). TxSubmissionTracker creates a bound callback and passes it to each manager.

- TxSubmitMgr constructor interface changes (new optional param)
- TxSubmitMgr owns event naming ‚Äî no translation layer
- Future transitions: author sees the pattern and knows to add a log call
- Emulator tests: don't pass callback ‚Äî zero impact

**Option B: Subscribe to `"transition"` event from TxSubmissionTracker** ‚Äî Zero TxSubmitMgr changes. TxSubmissionTracker maps internal transition names to user-facing event names.

- Requires mapping object translating TxSubmitMgr's internal transition vocabulary
- Implicit coupling: TxSubmissionTracker depends on transition name vocabulary with no compile-time enforcement
- Less granular: some semantic events (e.g. "backoff with N ms delay") happen within transition handlers, not as separate transitions
- Silent omission risk when new transitions are added

**Option C: Override `transition()` in TxSubmitMgr** ‚Äî Every transition logged automatically via override.

- No risk of omission but less control over event names
- Coupling to StateMachine internals
- Still needs constructor callback for routing

**Decision**: Option A.

**Rationale**: Two incisive factors:
1. **Semantic clarity at source** ‚Äî TxSubmitMgr is where the meaning lives. Human-readable event names and relevant detail strings (error messages, backoff intervals) are naturally available at each call site, not reconstructable from state machine metadata.
2. **Explicit over automatic** ‚Äî The log is for humans reading it after reload. `"notOk ‚Üí submitting"` is useless; `"submit-failed: rate limited"` is useful. That translation requires judgment at each call site ‚Äî the ~8-10 explicit calls are a feature, not a cost.

### Design Decision: Tracker Registration Path (Topic 3.2)

**Intent**: When a `TxSubmissionTracker` starts submitting, it must be registered in `TxBatcher._trackerRegistry` so the UI can look up live trackers by txHash. The question: how does the registration call get threaded from the tracker (which knows its txHash after signing) to the batcher (which owns the registry)?

**Key constraint**: `TxSubmissionTracker` calls `$signAndSubmit()` on itself via its own `onEntry.signingSingle` handler. `BatchSubmitController` never awaits it ‚Äî it just transitions the tracker's state machine. So `BatchSubmitController` has no direct point where it knows signing completed and `tracker.txId` is available.

**Options evaluated**:

**Option A: Pass batcher reference through the chain** ‚Äî `TxBatcher` ‚Üí `BatchSubmitController` ‚Üí `TxSubmissionTracker`. Tracker calls `batcher.registerTracker()` after signing. Two constructor interfaces change. Circular coupling: batcher creates controller creates tracker, tracker references batcher.

**Option B: Pass a `registerTracker` callback through the chain** ‚Äî Same threading as A but with a narrow callback instead of full object reference. Two constructor interfaces change. No type coupling to `TxBatcher`.

**Option C: `BatchSubmitController` registers after signing** ‚Äî Only one constructor interface changes. But `BatchSubmitController` doesn't await `$signAndSubmit()` ‚Äî it fires via the tracker's own state machine entry handler. No clean point to read `tracker.txId`. Doesn't work as described.

**Option D: Event-driven via existing infrastructure** ‚Äî `TxBatcher` already receives each `BatchSubmitController` via its own `"rotated"` event (including lazily-created ones from `get current()`). `BatchSubmitController` already emits `"txAdded"` with each `TxSubmissionTracker`. The tracker emits `"state:entered"` when entering `"submitting"` ‚Äî at which point `tracker.txId` is set after signing. Chain: `"rotated"` ‚Üí `"txAdded"` ‚Üí `"state:entered"` = `"submitting"` ‚Üí register.

**Option E: No registry ‚Äî walk controllers' `$txStates`** ‚Äî `findTracker` scans `current.$txStates` and `previous.$txStates`. But `$txStates` is keyed by internal batch ID, not txHash. Would need a scan or secondary index. Couples `TxBatcher.findTracker` to `BatchSubmitController` internals.

**Decision**: Option D.

**Rationale**: Zero constructor changes. Zero new parameters. All existing event infrastructure. The subscription is set up once in `TxBatcher`'s constructor ‚Äî `"rotated"` catches every `BatchSubmitController` (including lazily-created ones), `"txAdded"` catches every tracker, and `"state:entered" === "submitting"` is the exact moment `tracker.txId` becomes available.

### Design Decision: Tracker Lifecycle Ownership (Topic 3.3)

**Intent**: When `BatchSubmitController.destroy()` runs (batch rotation), it currently destroys all trackers. Trackers registered in `TxBatcher._trackerRegistry` must survive rotation because the UI still needs them.

**Options evaluated**:

**Option A: Boolean flag on tracker** ‚Äî `_preserveOnBatchDestroy` or similar. `BatchSubmitController.destroy()` checks flag before destroying. Simple but binary ‚Äî no information about *who* owns it.

**Option B: Predicate callback on `BatchSubmitController`** ‚Äî Batcher passes `shouldPreserve` callback at construction. Flexible but changes constructor interface for a single consumer.

**Option C: `"destroying"` event** ‚Äî Controller emits before destroying, batcher rescues trackers. Fragile ordering, synchronous rescue requirement.

**Option D: Remove tracker destruction from controller entirely** ‚Äî Trackers never explicitly destroyed. But they have timers and subscriptions that would leak.

**Decision**: Explicit ownership tag: `_ownedBy: "batch" | "globalTracker" = "batch"`. Set to `"globalTracker"` by `TxBatcher.registerTracker()`. `BatchSubmitController.destroy()` only destroys trackers where `_ownedBy === "batch"`. Extensible if a third owner emerges. Default matches current behavior.
