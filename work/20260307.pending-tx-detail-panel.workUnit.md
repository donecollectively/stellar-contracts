## Problem / Context

The `PendingTxTracker` shows compact dots for in-flight transactions but provides no drill-down capability — clicking a dot does nothing. Users can only see details during the active batch session via `TxBatchViewer`, but once the batch is rotated or the page is reloaded, all diagnostic information (build transcript, tx structure, submitter state) is lost.

Three connected problems:

1. **No persistent diagnostics**: The submission pipeline (`TxSubmitMgr`) tracks rich state (submit attempts, confirmations, backoff, errors) but only in memory. None of this survives page reload.

2. **Tracker lifecycle tied to batch**: `TxSubmissionTracker` objects are owned by `BatchSubmitController` and destroyed when the batch rotates. But the transactions they manage are still in-flight — the trackers should outlive the batch.

3. **No detail-on-demand from PendingTxTracker**: The dots have no onClick behavior. The existing `ShowTxDescription` component in `TxBatchViewer` has transcript/structure/diagnostics tabs, but it's private and requires live objects.

### Design

The solution has four parts:

**A. Persistent diagnostic fields on PendingTxEntry**: Capture `buildTranscript` (logger history), `txStructure` (dumpAny output), and `submissionLog` (incremental event log) at registration/signing time. These survive in Dexie across page reloads.

**B. Submission log written incrementally**: `TxSubmitMgr` writes `SubmissionLogEntry` events on each state transition via a callback. Events are appended to the Dexie `PendingTxEntry.submissionLog` via `appendSubmissionLog()`.

**C. Tracker lifecycle transfer to TxBatcher**: When a `TxSubmissionTracker`'s tx is signed and submission begins, the tracker is registered with `TxBatcher` by txHash. `TxBatcher` owns its lifecycle from that point — the tracker survives batch rotation and is destroyed only at terminal state (confirmed-certain or rolled-back).

**D. Shared TxDetailPanel with hybrid data sourcing**: Refactor `ShowTxDescription` → exported `TxDetailPanel`. `PendingTxTracker` onClick opens it, looking up a live tracker via `txBatcher.findTracker(txHash)` when available, falling back to Dexie data when not.

### TxBatcher — Tracker Registry

**REQT/s92rdnpzm7 (Tracker Registry)** — NEXT/draft: `TxBatcher` MUST maintain an in-memory `Map<txHash, TxSubmissionTracker>` for trackers that have been handed off from `BatchSubmitController`. MUST provide `findTracker(txHash: string): TxSubmissionTracker | undefined` for consumers (e.g. `PendingTxTracker`) to look up live tracker objects. MUST provide `registerTracker(txHash: string, tracker: TxSubmissionTracker): void` for `TxSubmissionTracker` to register itself when submission begins. The registry is session-scoped — not persisted, empty after page reload.

**REQT/37h2wenjy2 (Session-Scoped Registry)** — NEXT/draft: `TxBatcher`'s tracker registry MUST be session-scoped — populated during the session and emptied on page reload. Trackers in terminal states (confirmed-certain, rolled-back) are inert (no active timers or subscriptions) and cheap to hold. No explicit terminal cleanup is required — the registry is bounded by per-session transaction count. The `PendingTxEntry` in Dexie holds the complete persisted record regardless of tracker lifecycle.

**REQT/46ttdvnkmp (Capture Build Transcript)** — NEXT/draft: When signing completes and submission is about to begin, `TxSubmissionTracker` MUST capture `tcx.logger.formattedHistory` as `buildTranscript` and `dumpAny(tx, networkParams)` as `txStructure`. Each entry MUST include `at` (epoch ms), `event` (string), `submitter` (submitter name), and `detail` (optional — error message or state info). ### Design Decision: TxSubmitMgr Submission Log Emission (Topic 2.1)

**Intent**: After page reload, the user can see what happened during submission — which submitters tried, what failed/succeeded, retries, errors — as a timestamped event log persisted in Dexie.

**Options evaluated**:

**Option A: Explicit callback on TxSubmitMgr constructor** — Add optional `onSubmissionLog` callback. TxSubmitMgr calls it at ~8-10 specific code points (`trySubmit()` → `"submit-attempt"`, `notSubmitted()` → `"submit-failed"`, `didConfirm()` → `"confirmed"`, etc.). TxSubmissionTracker creates a bound callback and passes it to each manager.

- TxSubmitMgr constructor interface changes (new optional param)
- TxSubmitMgr owns event naming — no translation layer
- Future transitions: author sees the pattern and knows to add a log call
- Emulator tests: don't pass callback — zero impact

**Option B: Subscribe to `"transition"` event from TxSubmissionTracker** — Zero TxSubmitMgr changes. TxSubmissionTracker maps internal transition names to user-facing event names.

- Requires mapping object translating TxSubmitMgr's internal transition vocabulary
- Implicit coupling: TxSubmissionTracker depends on transition name vocabulary with no compile-time enforcement
- Less granular: some semantic events (e.g. "backoff with N ms delay") happen within transition handlers, not as separate transitions
- Silent omission risk when new transitions are added

**Option C: Override `transition()` in TxSubmitMgr** — Every transition logged automatically via override.

- No risk of omission but less control over event names
- Coupling to StateMachine internals
- Still needs constructor callback for routing

**Decision**: Option A.

**Rationale**: Two incisive factors:
1. **Semantic clarity at source** — TxSubmitMgr is where the meaning lives. Human-readable event names and relevant detail strings (error messages, backoff intervals) are naturally available at each call site, not reconstructable from state machine metadata.
2. **Explicit over automatic** — The log is for humans reading it after reload. `"notOk → submitting"` is useless; `"submit-failed: rate limited"` is useful. That translation requires judgment at each call site — the ~8-10 explicit calls are a feature, not a cost.

### Design Decision: Tracker Registration Path (Topic 3.2)

**Intent**: When a `TxSubmissionTracker` starts submitting, it must be registered in `TxBatcher._trackerRegistry` so the UI can look up live trackers by txHash. The question: how does the registration call get threaded from the tracker (which knows its txHash after signing) to the batcher (which owns the registry)?

**Key constraint**: `TxSubmissionTracker` calls `$signAndSubmit()` on itself via its own `onEntry.signingSingle` handler. `BatchSubmitController` never awaits it — it just transitions the tracker's state machine. So `BatchSubmitController` has no direct point where it knows signing completed and `tracker.txId` is available.

**Options evaluated**:

**Option A: Pass batcher reference through the chain** — `TxBatcher` → `BatchSubmitController` → `TxSubmissionTracker`. Tracker calls `batcher.registerTracker()` after signing. Two constructor interfaces change. Circular coupling: batcher creates controller creates tracker, tracker references batcher.

**Option B: Pass a `registerTracker` callback through the chain** — Same threading as A but with a narrow callback instead of full object reference. Two constructor interfaces change. No type coupling to `TxBatcher`.

**Option C: `BatchSubmitController` registers after signing** — Only one constructor interface changes. But `BatchSubmitController` doesn't await `$signAndSubmit()` — it fires via the tracker's own state machine entry handler. No clean point to read `tracker.txId`. Doesn't work as described.

**Option D: Event-driven via existing infrastructure** — `TxBatcher` already receives each `BatchSubmitController` via its own `"rotated"` event (including lazily-created ones from `get current()`). `BatchSubmitController` already emits `"txAdded"` with each `TxSubmissionTracker`. The tracker emits `"state:entered"` when entering `"submitting"` — at which point `tracker.txId` is set after signing. Chain: `"rotated"` → `"txAdded"` → `"state:entered"` = `"submitting"` → register.

**Option E: No registry — walk controllers' `$txStates`** — `findTracker` scans `current.$txStates` and `previous.$txStates`. But `$txStates` is keyed by internal batch ID, not txHash. Would need a scan or secondary index. Couples `TxBatcher.findTracker` to `BatchSubmitController` internals.

**Decision**: Option D.

**Rationale**: Zero constructor changes. Zero new parameters. All existing event infrastructure. The subscription is set up once in `TxBatcher`'s constructor — `"rotated"` catches every `BatchSubmitController` (including lazily-created ones), `"txAdded"` catches every tracker, and `"state:entered" === "submitting"` is the exact moment `tracker.txId` becomes available.

### Design Decision: Tracker Lifecycle Ownership (Topic 3.3)

**Intent**: When `BatchSubmitController.destroy()` runs (batch rotation), it currently destroys all trackers. Trackers registered in `TxBatcher._trackerRegistry` must survive rotation because the UI still needs them.

**Options evaluated**:

**Option A: Boolean flag on tracker** — `_preserveOnBatchDestroy` or similar. `BatchSubmitController.destroy()` checks flag before destroying. Simple but binary — no information about *who* owns it.

**Option B: Predicate callback on `BatchSubmitController`** — Batcher passes `shouldPreserve` callback at construction. Flexible but changes constructor interface for a single consumer.

**Option C: `"destroying"` event** — Controller emits before destroying, batcher rescues trackers. Fragile ordering, synchronous rescue requirement.

**Option D: Remove tracker destruction from controller entirely** — Trackers never explicitly destroyed. But they have timers and subscriptions that would leak.

**Decision**: Explicit ownership tag: `_ownedBy: "batch" | "globalTracker" = "batch"`. Set to `"globalTracker"` by `TxBatcher.registerTracker()`. `BatchSubmitController.destroy()` only destroys trackers where `_ownedBy === "batch"`. Extensible if a third owner emerges. Default matches current behavior.

### Design Decision: PendingTxTracker ↔ TxBatcher Connection (Topic 4.3)

**Intent**: `PendingTxTracker` needs access to live `TxSubmissionTracker` objects for drill-down detail. The batcher holds the registry. How does the connection work?

**Context**: The actual consumer (`MemberStatusPanel` in s3gov) already has `capo.setup.txBatcher` via `useCapoDappProvider()`. `PendingTxTracker` is rendered directly inside it.

**Options evaluated**:

**Option 1: `findTracker` callback prop** — Narrow function prop. `PendingTxTracker` doesn't know about `TxBatcher`. Minimal coupling but hides origin.

**Option 2: `useCapoDappProvider` inside `PendingTxTracker`** — Zero prop changes but hidden context dependency. Implicit coupling — Code Whisperer red flag.

**Option 3: `txBatcher` prop** — Pass the real object. Wider than strictly needed but honest about what it is. If `PendingTxTracker` ever needs more from the batcher, no interface change needed.

**Decision**: Option 3. The interface segregation concern is theoretical — we own both sides. Passing the real object is cleaner than wrapping it in a function.

## Post-Completion Refinements

### UI Polish (iterative with user)
- **Submission log styling**: Changed from card-style (bordered, padded) to log-style (compact mono rows, no borders/margins). Added scrollable container.
- **Structure tab sizing**: Reduced to ~80% of original (`text-[1.05em]/4` from `text-[1.30em]/4.5`).
- **Copy-to-clipboard**: Added click-to-copy on signed and unsigned CBOR hex with floating toast confirmation (portaled to `document.body`). Shift+click skips copy for text selection. Also added click-to-copy on tx hash in `PendingTxTracker` header.
- **Backoff log formatting**: Human-readable seconds (`23.1s`) instead of raw milliseconds (`23126.8585849ms`) in `TxSubmitMgr.emitLog()`.
- **Removed confirm-attempt log event**: Redundant — `confirmed` or `not-confirmed` follows immediately.
- **Selection indicator**: Replaced glow effect with downward triangle `▼` under selected dot, fixed height to prevent layout shift.
- **Panel header**: Added zoom `⛶` and close `✕` buttons matching s3gov log panel pattern. Double-click header to toggle zoom.
- **Panel portaling**: Detail panel portaled to `document.body` to escape parent stacking contexts.
- **Zoom behavior**: Zoomed panel stays at same vertical position (anchored to dots container), expands to full viewport width, fills remaining height.
- **Scroll consolidation**: Removed inner `max-h-*` constraints from `TxDetailPanel` tab content; outer container handles scrolling. `TxBatchViewer` card gets `max-h-[90vh] overflow-y-auto`.
- **Removed redundant close button**: `TxDetailPanel` no longer renders its own close button — parent contexts (`PendingTxTracker`, `TxBatchViewer`) provide their own.
- **Removed redundant tx name**: `TxDetailPanel` shows only `moreInfo`, not `description` or `txName` (parent headers already display the name).

### Diagnostics Reorganization
- **Submitter status moved**: Live `TxSubmitMgr` status cards moved from transcript tab to diagnostics tab.
- **Signed tx structure removed**: Removed `dumpAny(signedTx)` rendering and `decodeTx` import — was erroring and not needed.
- **Live submission log**: `TxSubmissionTracker` now accumulates `liveSubmissionLog` entries in-memory alongside Dexie persistence. `TxDetailPanel` reads from live array when tracker is active, falls back to persisted data after reload.

### Data Fixes
- **`txName` on addlTxns**: `StellarTxnContext.includeAddlTxn()` now sets `txName` from the `txnName` parameter (previously only used as the map key). Fixes `txName: undefined` on persisted `PendingTxEntry` records.
- **`moreInfo` passthrough**: `TxSubmissionTracker.$signAndSubmit()` now passes `txd.moreInfo` to `registerPendingTx()` (was missing).
