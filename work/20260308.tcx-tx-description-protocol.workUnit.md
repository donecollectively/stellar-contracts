## Problem / Context

Transaction descriptions shown to users in the TxBatchViewer have three display tiers:

- **txName** — bold primary label
- **description** — dimmed secondary text below txName
- **moreInfo** — italic tertiary text, shown in TxDetailPanel advanced view only

Currently, `txName` and `description` both default to `tcx.txnName`, so they always show the same string. There is no way for a controller's `mkTxnXYZ` method to set name and description independently. Worse, the UI layer can override controller-set labels by passing `addlTxInfo` to `buildAndQueueAll()` / `submitAll()`, subverting the domain API's intent.

The goal is to let the onchain module's controller layer (the dAPI) establish authoritative tx labels that downstream UI code cannot silently override.

### Current flow

```
mkTcx("some name")           → sets tcx.txnName
  │
  └─ buildAndQueueAll(options?)
       │  
       │  options.addlTxInfo?: { txName?, description }
       │  
       │  defaults when addlTxInfo not provided:
       │      txName:      tcx.txnName
       │      description: tcx.txnName  (same source — always duplicated)
       │  
       └─► TxDescription { txName, description, moreInfo }
              │
              └─► TxBatchViewer renders:
                    <b>{txName}</b>
                    <div opacity-50>{description}</div>
                    (moreInfo in detail panel only)
```

Both `txName` and `description` come from one string. The UI can override them via `addlTxInfo`. Controllers have no way to lock their labels.

## Plan / Guidance

### 1. Add `tcx.withDescription()` and `tcx.withMoreInfo()`

New chainable methods on `StellarTxnContext` to set description and moreInfo independently of the name:

```ts
tcx.withDescription(desc: string): this
tcx.withMoreInfo(info: string): this
```

These store values on the tcx instance. When `buildAndQueueAll` / `submitAll` constructs `TxDescription`, it uses `tcx.description` (falling back to `tcx.txnName` if unset) and `tcx.moreInfo`.

### 2. Expand `mkTcx()` to accept an options object

```ts
mkTcx(name: string): StellarTxnContext              // existing
mkTcx(opts: { name: string, description?: string, moreInfo?: string }): StellarTxnContext  // new
```

When the object form is used, calls `withName`, `withDescription`, `withMoreInfo` as appropriate.

### 3. Add `tcxWithGuardedTxDetails(forcedTcx, initialTcx?)` utility

For use by `mkTxnXYZ` methods that accept an optional initial tcx from callers:

- MUST override `name` and `description` on the initial tcx from the forced tcx
- MAY provide its own `moreInfo` in the forced tcx
- MUST use forced tcx's `moreInfo` unless initial tcx has its own `moreInfo`
- The initial tcx MUST NOT have `lockedLabels`; the returned tcx MUST have `lockedLabels`
- Returns the initial tcx (or forced tcx if no initial) with labels applied and locked

### 4. Add `lockedLabels` flag on tcx

When `lockedLabels` is true:
- `withName()`, `withDescription()`, `withMoreInfo()` warn and no-op
- `submitAll()` / `buildAndQueueAll()` ignore `addlTxInfo.txName` and `addlTxInfo.description` from options, using the tcx's locked values instead. Log a warning if the caller attempted to override.

### 5. Deprecate `buildAndQueueAll()` at the API level

Redirect callers to `submitAll()`. `buildAndQueueAll()` continues to work but logs a deprecation warning.

### 6. Restrict `addlTxInfo` on `submitAll()`

The type for `addlTxInfo` on `SubmitOptions` should not allow `txName` or `description` — only `moreInfo` and other supplementary fields. Runtime warns if callers pass `txName` or `description`, directing them to use `tcx.withName()` / `tcx.withDescription()` instead.

### 7. Update `mkTxnXYZ` methods to use guarded details

Controller methods like `mkTxnUpdateRecord`, `mkTxnCreateRecord`, and domain-specific builders (e.g. `mkTxnAddToMarketSale`) should use `tcxWithGuardedTxDetails()` to establish their labels, preventing UI callers from overriding them.

## Acceptance Criteria

- [ ] `tcx.withDescription()` and `tcx.withMoreInfo()` set independent fields
- [ ] `mkTcx()` accepts `{ name, description, moreInfo }` object form
- [ ] `tcxWithGuardedTxDetails(forcedTcx, initialTcx?)` correctly overrides name/description, preserves caller moreInfo, and locks labels
- [ ] `lockedLabels` prevents downstream overrides of name/description, with warnings
- [ ] `buildAndQueueAll()` logs deprecation, redirects to `submitAll()`
- [ ] `submitAll()` type does not allow `txName`/`description` in `addlTxInfo`; runtime warns
- [ ] `TxBatchViewer` renders all three tiers correctly when all three are set
- [ ] Existing callers of `buildAndQueueAll()` continue to work (deprecation warning only)
- [ ] At least one `mkTxnXYZ` method demonstrates the guarded pattern
