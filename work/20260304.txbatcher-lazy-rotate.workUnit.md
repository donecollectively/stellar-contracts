# Work Unit: txbatcher-lazy-rotate

**Title**: Simplify TxBatcher.rotate() to lazy chainBuilder/BSC initialization
**Created**: 2026-03-04
**Source**: ad-hoc — code review during TxSubmissionTracker fix identified that rotate() eagerly creates a new chainBuilder and BatchSubmitController with a spread-copy of setup, detaching BSC's setup from the shared reference. The `current` getter already has correct lazy initialization logic that creates both and shares the setup reference properly.

## Problem / Context

`TxBatcher.rotate()` currently:
1. Creates a new `TxChainBuilder` eagerly
2. Sets `this.setup.chainBuilder = chainBuilder` (current unstaged diff)
3. Creates a new `BatchSubmitController` with `setup: { ...this.setup, chainBuilder }` — a **spread copy** that detaches the BSC's setup from the shared `this.setup` reference

This spread-copy pattern is problematic: when `StellarTxnContext` later sets `this.setup.chainBuilder`, the BSC doesn't see the update because it has its own copy. By contrast, the `current` getter creates BSC with `setup: this.setup` (shared reference) and handles chainBuilder creation correctly.

The fix: make `rotate()` null out both `_current` and `setup.chainBuilder`, letting the `current` getter's lazy initialization handle creation on next access. This is cleaner and ensures the BSC always shares the setup reference.

**Coordinated change required**: The `StellarTxnContext` constructor (lines 424-438) captures `currentBatch` before calling `rotate()`, then uses the stale reference for chainBuilder backfill. After rotate nulls `_current`, the constructor must re-access `txBatcher.current` (triggering lazy init) rather than relying on the pre-rotate `currentBatch` reference.

The `cancel()` method has the same spread-copy pattern and should be evaluated for the same treatment.

## Plan / Guidance

### Step 1: Simplify `rotate()`

Change `rotate()` to:
```typescript
rotate() {
    if (!this.setup) throw new Error(`setup not set`);
    if (!this.signingStrategy) throw new Error(`signingStrategy not set`);
    if (!this.canRotate()) throw new Error(`must verify canRotate() before rotating`);

    this.previous?.destroy();
    this.previous = this._current;  // direct field access, not getter
    this._current = undefined;
    this.setup.chainBuilder = undefined;
}
```

Key details:
- Use `this._current` (not `this.current`) to avoid triggering lazy init just to stash the old BSC
- Remove `newChainBuilder` parameter — no longer needed
- The `"rotated"` event will fire lazily on next `current` access (no external subscribers exist currently)
- Use `undefined` rather than `null` to match the optional type (`_current?: BatchSubmitController`)

### Step 2: Fix StellarTxnContext constructor

After `rotate()` nulls everything, the constructor must re-access `txBatcher.current` to trigger lazy init. The post-rotate backfill block (lines 430-438) becomes unnecessary since the getter handles chainBuilder creation. Simplify to:

```typescript
const currentBatch = this.currentBatch;
if (!currentBatch || currentBatch.isConfirmationComplete) {
    this.setup.txBatcher.rotate();  // remove chainBuilder arg
}
// Re-access after potential rotate to get the fresh BSC
// The getter creates chainBuilder and sets this.setup.chainBuilder
const activeBatch = this.currentBatch;
```

The explicit chainBuilder backfill block can be removed — the `current` getter handles it.

### Step 3: Evaluate `cancel()` method

`cancel()` has the same spread-copy pattern. Apply the same treatment: null out `_current` and `chainBuilder`, let the getter handle lazy init. This also removes the redundant BSC creation from `cancel()`.

### Step 4: Remove `newChainBuilder` parameter from `rotate()` signature

The parameter is no longer needed. Check that the single caller (`StellarTxnContext` constructor) no longer passes it.
