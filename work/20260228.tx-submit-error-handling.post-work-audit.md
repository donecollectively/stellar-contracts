## Findings

### Finding: f1 — Silent terminal failure when rescue budget exhausts

**Status**: resolved
**Reviewer**: Code Whisperer
**Severity**: Concern
**Location**: In `onEntry.failed` handler (bounded rescue else-branch) at `src/networkClients/TxSubmitMgr.ts:718`

**Observation**: When `isBadTx` is set and `rescueAttempts` exceeds `maxRescueAttempts`, the handler does nothing — no log, no event, no state change. The state machine silently stops in `failed` with no indication to consumers or operators that it has permanently given up. The `$$statusSummary` getter exposes `deferredAction` (which will be empty) and `isBadTx` (which is set), but there's no explicit "permanently failed" signal. In production debugging, the difference between "still trying" and "permanently given up" is invisible without inspecting `rescueAttempts` directly.

**Resolution**: Added `this.log()` call in the else branch: `"reached maximum rescue attempts for transaction (N/5)"`. All tests pass.

### Finding: f2 — `resubmit` transition reachability

**Status**: resolved — finding was inaccurate; implementation already wires the path
**Reviewer**: Code Whisperer
**Severity**: Concern (retracted)
**Location**: In `onEntry.failed` handler and `checkTxInputStatus` stub at `src/networkClients/TxSubmitMgr.ts:522,738-755,1003-1007`

**Observation (corrected)**: The original finding was based on the pre-implementation analysis in the work unit Advisory Notes, which stated the `!successfulSubmitAt` branch was unreachable. The actual committed implementation diverges: `onEntry.failed` now has a three-branch dispatch where the middle branch (`!successfulSubmitAt`) fires the `resubmit` transition. This branch activates when `checkTxInputStatus()` returns `"premature"` — indicating tx inputs aren't visible yet rather than permanently spent. Currently `checkTxInputStatus` is a stub returning `"bad"` (with a TODO), so the `resubmit` path doesn't fire in practice. But the wiring is structurally complete: stub method → notSubmitted classification → onEntry.failed branch → resubmit transition. Once the stub is implemented, all branches become live.

The work unit's Advisory Notes (Pre-Work Testing Assessment) and test file comment about the dropped `direct-resubmit-from-failed` test are now stale — they describe a design decision that was reversed during implementation.

**Resolution**: The finding was inaccurate. The implementation already wires `resubmit` through a coherent staged design. The stale Advisory Notes and test comment describing the branch as unreachable should be updated when the `checkTxInputStatus` detection logic is implemented.

### Finding: f3 — `$$statusSummary` doesn't expose `rescueAttempts`

**Status**: open
**Reviewer**: Architect
**Severity**: Concern
**Location**: In `$$statusSummary` getter at `src/networkClients/TxSubmitMgr.ts:155-191`

**Observation**: The new `rescueAttempts` counter is added to `SubmitManagerState` and used in `onEntry.failed`, but it is not exposed through `$$statusSummary` — the primary interface through which `TxSubmissionTracker` and UI consumers observe submitter state. Consumers can see `isBadTx` and `status: "failed"` but cannot distinguish "rescue in progress (attempt 3/5)" from "rescue exhausted (permanently failed)." The `deferredAction` field provides indirect evidence (empty when exhausted) but is not a clear signal. This limits the upstream `updateSubmitterState` aggregation and any UI that renders submission progress.
