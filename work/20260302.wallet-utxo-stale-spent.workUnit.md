## Problem / Context

`CachedUtxoIndex.getUtxos(walletAddress)` returns UTxOs that have already been spent on-chain. Observed: 43 UTxOs totaling ~5455 ADA for a wallet actually holding ~653.5 ADA. Stale spent UTxOs inflate the total, causing incorrect balance calculations and coin selection across any code path relying on `wallet.utxos`.

### Root Cause

For the **Capo address**, spent UTXO eviction works correctly: `processTransactionForNewUtxos()` examines `tx.body.inputs` and calls `markUtxoSpent()` for each consumed UTXO (REQT/hhbcnvd9aj). This transaction-level processing ensures the cache reflects actual on-chain state.

For **wallet addresses**, no such reconciliation exists. Both `addWalletAddress()` (initial registration) and `syncWalletAddressIfStale()` (periodic refresh) follow the same pattern:

1. Fetch fresh UTxOs from network via `this.network.getUtxos(heliosAddress)`
2. Store each via `this.store.saveUtxo(entry)` — this is a put-by-PK, so existing entries with the same `utxoId` are overwritten
3. Update the wallet address sync state

**The critical gap**: Step 2 adds or overwrites UTxOs that still exist on-chain, but **never identifies or marks UTxOs that are no longer in the network response**. A UTXO that was cached during a previous sync but has since been spent on-chain remains in the store with `spentInTx: null`. Since `findUtxosByAddress()` returns all entries where `spentInTx` is null (REQT/g3jen1rcvd), these stale entries are returned alongside current ones.

### Why the Capo path doesn't have this problem

The Capo address is monitored via `checkForNewTxns()` → `processTransactionForNewUtxos()`, which processes each transaction individually:
- New outputs are indexed via `indexUtxoFromOutput()`
- Spent inputs are marked via `markUtxoSpent(utxoId, txHash)`

This transaction-level approach keeps `spentInTx` accurate. Wallet addresses use a snapshot-based approach (fetch all current UTxOs) that discards the "what changed" information needed to mark spent entries.

### Impact

Any consumer calling `getUtxos()` with a wallet address gets inflated results:
- **Balance calculations** show more ADA than actually available
- **Coin selection** may attempt to spend already-consumed UTxOs, causing transaction failures
- **The problem compounds over time** — each sync adds current UTxOs but never removes stale ones, so the cache grows monotonically

## Plan / Guidance

### Fix: Snapshot-based reconciliation for wallet address syncs

Both `addWalletAddress()` and `syncWalletAddressIfStale()` need a reconciliation step after fetching fresh UTxOs from the network. The approach:

1. **Fetch fresh UTxOs** from network (existing step)
2. **Build a set** of `utxoId`s from the network response
3. **Query existing cached UTxOs** for this address via `store.findUtxosByAddress(address)` — this returns only unspent entries (where `spentInTx` is null)
4. **Identify stale entries**: cached UTxOs whose `utxoId` is NOT in the fresh set
5. **Mark stale entries as spent**: call `store.markUtxoSpent(utxoId, "reconciled")` for each stale entry — using `"reconciled"` (or similar sentinel) as the `spentInTx` value since there's no specific spending tx hash available in this path
6. **Store fresh UTxOs** (existing step)

**Implementation detail**: The sentinel value for `spentInTx` matters for the in-flight transaction integration. The pending-tx rollback path clears `spentInTx` where it matches a specific txHash. A sentinel like `"reconciled"` won't collide with real txHashes, so rollback won't accidentally restore reconciliation-evicted UTxOs. This is correct behavior.

**Alternative considered**: Delete stale entries entirely instead of soft-deleting. Rejected because the existing `spentInTx` soft-delete pattern (REQT/11msfc4wv8) is established for good reasons (debugging, chain reorg handling). Wallet address UTxOs should follow the same convention.

### Store interface consideration

The reconciliation logic can be implemented entirely in `CachedUtxoIndex` using existing store methods (`findUtxosByAddress`, `markUtxoSpent`, `saveUtxo`). No new store interface methods are strictly required, though a `markStaleUtxosByAddress(address, freshUtxoIds)` convenience method on the store could encapsulate the pattern.

### Shared reconciliation helper

Both `addWalletAddress()` and `syncWalletAddressIfStale()` need identical reconciliation logic. Extract a private method (e.g., `reconcileWalletUtxos(address, freshUtxos)`) that:
1. Builds the fresh utxoId set
2. Queries existing cached unspent UTxOs for the address
3. Marks stale entries as spent
4. Stores fresh UTxOs

Both callers delegate to this method instead of duplicating the loop.

#### Architect leads — findings

##### Finding F1: Reconciliation sentinel value needs architectural alignment with spentInTx semantics

**Severity**: Concern

The plan proposes using `"reconciled"` as the `spentInTx` sentinel for snapshot-evicted wallet UTxOs. This introduces a new semantic category for `spentInTx` — currently it's either `null` (unspent) or a txHash (spent by a specific transaction). The architecture doc (REQT/11msfc4wv8) defines `spentInTx` as "the spending transaction's hash" — a sentinel string breaks that contract.

Downstream implications:
- The in-flight tx rollback path (`clearSpentByTx`) queries `WHERE spentInTx === txHash` — a sentinel won't collide, which is correct
- But any future code that assumes `spentInTx` is always a valid txHash (e.g., for chain reorg detection, spend-graph analysis) would break
- REQT/11msfc4wv8 says "This preserves historical data for debugging" — a sentinel loses the "which tx spent this" information entirely

Two alternatives worth considering:
1. **Hard-delete for wallet UTxOs only** — wallet UTxOs don't need the same audit trail as Capo UTxOs. The architecture distinguishes them (REQT/620ypcc34d). Hard-deleting stale wallet entries avoids polluting the `spentInTx` semantics entirely.
2. **Sentinel prefix convention** — if soft-delete is preferred, use a prefix like `reconciled:` so code can distinguish sentinels from txHashes programmatically (txHashes are hex, sentinels start with alpha-colon).

#### Code Whisperer — sequenced follow-up

##### Finding F2: Duplicate sync logic — both methods share identical fetch-store-update pattern

**Severity**: Concern

`addWalletAddress()` (lines 616-650) and `syncWalletAddressIfStale()` (lines 660-700) share the exact same fetch→store→update loop. The work unit's plan correctly identifies this and proposes a `reconcileWalletUtxos()` helper. This finding reinforces the plan and adds structural precision.

The shared method should encapsulate the complete cycle: fetch from network → build fresh ID set → query existing cached unspent → mark stale as spent → store fresh → update wallet sync state. Both callers should reduce to: guard check + call shared method.

The existing code already has a structural seam: `txInputToIndexEntry()` is the conversion point. The shared helper should use this consistently.

##### Finding F3: `addWalletAddress` re-registration path skips reconciliation

**Severity**: Concern

`addWalletAddress()` returns early when the address is already registered (`if (existing) return`). But a re-registration call is the natural recovery path when a user suspects stale data — reconnecting their wallet calls `addWalletAddress` again. With the current guard, re-registration silently does nothing. The user's stale data persists.

Two resolution options:
1. Remove the early return for already-registered addresses — always reconcile. This makes `addWalletAddress` idempotent in effect (safe to call multiple times, always produces correct state).
2. Keep the guard but provide a separate `resyncWalletAddress(address)` public method that forces reconciliation regardless of staleness timer.

Option 1 is simpler and aligns with the REQT/mp4dx7ngvf requirement ("Must trigger an initial sync for newly registered addresses") — arguably a re-registration SHOULD re-sync. The "skip if already registered" guard was a performance optimization that becomes incorrect once reconciliation is needed.
