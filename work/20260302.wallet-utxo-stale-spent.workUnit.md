## Problem / Context

`CachedUtxoIndex.getUtxos(walletAddress)` returns UTxOs that have already been spent on-chain. Observed: 43 UTxOs totaling ~5455 ADA for a wallet actually holding ~653.5 ADA. Stale spent UTxOs inflate the total, causing incorrect balance calculations and coin selection across any code path relying on `wallet.utxos`.

### Root Cause

For the **Capo address**, spent UTXO eviction works correctly: `processTransactionForNewUtxos()` examines `tx.body.inputs` and calls `markUtxoSpent()` for each consumed UTXO (REQT/hhbcnvd9aj). This transaction-level processing ensures the cache reflects actual on-chain state.

For **wallet addresses**, no such reconciliation exists. Both `addWalletAddress()` (initial registration) and `syncWalletAddressIfStale()` (periodic refresh) follow the same pattern:

1. Fetch fresh UTxOs from network via `this.network.getUtxos(heliosAddress)`
2. Store each via `this.store.saveUtxo(entry)` — this is a put-by-PK, so existing entries with the same `utxoId` are overwritten
3. Update the wallet address sync state

**The critical gap**: Step 2 adds or overwrites UTxOs that still exist on-chain, but **never identifies or marks UTxOs that are no longer in the network response**. A UTXO that was cached during a previous sync but has since been spent on-chain remains in the store with `spentInTx: null`. Since `findUtxosByAddress()` returns all entries where `spentInTx` is null (REQT/g3jen1rcvd), these stale entries are returned alongside current ones.

### Why the Capo path doesn't have this problem

The Capo address is monitored via `checkForNewTxns()` → `processTransactionForNewUtxos()`, which processes each transaction individually:
- New outputs are indexed via `indexUtxoFromOutput()`
- Spent inputs are marked via `markUtxoSpent(utxoId, txHash)`

This transaction-level approach keeps `spentInTx` accurate. Wallet addresses use a snapshot-based approach (fetch all current UTxOs) that discards the "what changed" information needed to mark spent entries.

### Impact

Any consumer calling `getUtxos()` with a wallet address gets inflated results:
- **Balance calculations** show more ADA than actually available
- **Coin selection** may attempt to spend already-consumed UTxOs, causing transaction failures
- **The problem compounds over time** — each sync adds current UTxOs but never removes stale ones, so the cache grows monotonically

## Plan / Guidance

### Change 1: Wallet transaction monitoring in `checkForNewTxns`

Extend `checkForNewTxns()` to poll wallet addresses alongside the Capo address. After processing Capo transactions (existing behavior), iterate over `store.getAllWalletAddresses()` and run the same per-address Blockfrost polling pattern for each:

1. Query `addresses/${walletAddr}/transactions?order=asc&count=${syncPageSize}&from=${walletEntry.lastBlockHeight + 1}`
2. Process each discovered transaction through the existing `processTransactionForNewUtxos()` — this marks spent inputs with real txHashes via `markUtxoSpent` and indexes new outputs
3. After processing, update the wallet entry's `lastBlockHeight` to the highest block seen

This gives wallet UTxOs the same correctness guarantees as Capo UTxOs: `spentInTx` is always a real txHash, no sentinels. The existing pagination and page-limit logic should apply per-address.

**Structural note**: The Capo address polling loop and wallet address polling loops share the same pattern. Consider extracting the address-polling logic into a helper that both the Capo path and wallet path can call.

### Change 2: Snapshot sync as secondary defense

`syncWalletAddressIfStale()` remains as the on-demand freshness path (triggered by `getUtxos` when the staleness timer expires). After fetching fresh UTxOs from the network:

1. Build a set of `utxoId`s from the network response
2. Query existing cached unspent UTxOs for this address
3. **Remove** cached entries whose `utxoId` is absent from the fresh set — these are stale cache entries, not audit-trail records
4. Store the fresh entries

This is a secondary defense: the primary eviction mechanism is Change 1 (transaction-based monitoring with real txHashes). The snapshot sync catches anything the transaction-based path missed (e.g., transactions that occurred before the wallet was registered, or edge cases in Blockfrost pagination).

### Change 3: `addWalletAddress` reconciliation

`addWalletAddress()` should apply the same snapshot reconciliation as Change 2 when the address is already registered. The current early-return guard (`if (existing) return`) silently skips sync. Instead, always run the snapshot sync — making `addWalletAddress` idempotent (safe to call multiple times, always produces correct state).

### Change 4: Unconditional slot advancement

At the end of `checkForNewTxns()`, call `fetchAndStoreLatestBlock()` unconditionally — regardless of whether any new transactions were discovered for any address. This ensures `lastSlot`/`now` always reflect the current chain tip. Currently `lastSlot` only advances when new transactions are found, causing stale `now` values and delayed pending-tx deadline rollbacks.

The `fetchAndStoreLatestBlock()` method already exists and handles the block fetch, store, and state update.
