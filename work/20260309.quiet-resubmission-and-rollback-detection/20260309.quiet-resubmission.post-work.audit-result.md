## Findings

#### Finding: r8kq2m4n5w — Rollback tx scan limited to same-height canonical blocks only
**Status**: open
**Location**: In `executeBlockRollback()` (builds canonicalTxSet by scanning canonical blocks for tx lists) at `src/networkClients/UtxoIndex/CachedUtxoIndex.ts:~1950-1970`

**Observation**: The canonical tx scanning loop iterates over `rolledBackBlockHashes` and for each, looks up the canonical block at the same height via `canonicalByHeight.get(rbBlock.height)`. It only fetches transaction lists from canonical blocks at heights where a rolled-back block exists. Per explicit stakeholder direction: "any of the blocks in the replacement fork are candidates for finding any tx that was on a losing fork." A tx from a rolled-back block can land at a different height on the winning fork. The current implementation would miss it and incorrectly revert to pending.

**Resolution**: The scanning loop should fetch transaction lists from all canonical blocks above the fork point — not just those at matching heights. The fork point is identifiable from `detectRolledBackBlocks` as the height just below the lowest rolled-back block. All canonical blocks at or above that height are replacement fork blocks and should be scanned. This includes blocks that replace rolled-back blocks AND any additional blocks the winning fork produced beyond the old fork's extent. Blocks below the fork point are identical on both chains and need not be scanned.
