## Findings

#### Finding: r8kq2m4n5w — Rollback tx scan limited to same-height canonical blocks only
**Status**: open
**Location**: In `executeBlockRollback()` (builds canonicalTxSet by scanning canonical blocks for tx lists) at `src/networkClients/UtxoIndex/CachedUtxoIndex.ts:~1950-1970`

**Observation**: The canonical tx scanning loop iterates over `rolledBackBlockHashes` and for each, looks up the canonical block at the same height via `canonicalByHeight.get(rbBlock.height)`. It only fetches transaction lists from canonical blocks at heights where a rolled-back block exists. Per explicit stakeholder direction: "any of the blocks in the replacement fork are candidates for finding any tx that was on a losing fork." A tx from a rolled-back block can land at a different height on the winning fork. The current implementation would miss it and incorrectly revert to pending.

**Resolution**: The scanning loop should fetch transaction lists from all canonical blocks above the fork point — not just those at matching heights. The fork point is identifiable from `detectRolledBackBlocks` as the height just below the lowest rolled-back block. All canonical blocks at or above that height are replacement fork blocks and should be scanned. This includes blocks that replace rolled-back blocks AND any additional blocks the winning fork produced beyond the old fork's extent. Blocks below the fork point are identical on both chains and need not be scanned.

**Coder fix reviewed** (commit `0b3a0d91`): Fork-point detection logic is correct — walks canonical blocks ascending, finds last shared height, scans everything above. Two residual concerns raised as incremental findings below.

#### Finding: xdvvjvyrrc — Redundant store queries for fork-point detection
**Status**: resolved
**Location**: In `executeBlockRollback()` (fork-point detection loop) at `src/networkClients/UtxoIndex/CachedUtxoIndex.ts:~1948-1960`, duplicating work done by `detectRolledBackBlocks()` at `~1898-1904`

**Observation**: Both `detectRolledBackBlocks` and `executeBlockRollback` walk the canonical blocks array comparing each canonical block hash against the stored block at the same height. `detectRolledBackBlocks` does this to identify rolled-back hashes; `executeBlockRollback` does it again to find the fork point. These are the same comparison — the fork point is implicit in the first function's results (the lowest rolled-back height minus one). The fork point height should be computed and returned from `detectRolledBackBlocks` alongside `rolledBackHashes`, eliminating the duplicate store queries in `executeBlockRollback`.

**Resolution**: Fixed in `9e504feb`. Fork point detection moved into `detectRolledBackBlocks` during the same loop that identifies rolled-back hashes. Uses `divergenceFound` flag to freeze `lastSharedHeight` after first mismatch. `forkHeight` returned alongside other results and passed to `executeBlockRollback`, which no longer queries the store for fork-point detection.

#### Finding: jd03an8y64 — Rollback tests don't exercise fork-point scanning logic
**Status**: resolved
**Location**: In rollback execution tests (`re-anchor`, `revert-to-pending`, `block-state`, `rollback-event`) at `src/networkClients/UtxoIndex/02g-BlockProcessing.test.ts:~1604-1710`

**Observation**: All rollback execution tests pass `[canonicalBlock101]` as the entire canonical blocks array — just the single replacement block, with no blocks below the fork point. The fork-point detection runs but `lastSharedHeight` ends up -1 (the only canonical block is the divergent one), so `forkHeight` = 0. Tests pass because the single canonical block IS scanned, but they don't validate that blocks below the fork point are excluded or that txs at different heights on the winning fork are found. The new scanning logic is functionally untested.

**Resolution**: Fixed in `9e504feb`. Tests now pass `fullCanonicalChain()` with blocks 98-102 (pre-fork 98-100 matching, fork at 101) and explicit `FORK_HEIGHT = 101`. Pre-fork blocks' `/txs` endpoints are deliberately NOT mocked — any attempt to scan them would trigger a mock error, providing an implicit assertion that fork-point exclusion works correctly.
