# Dismissed Concerns: Quiet Resubmission Post-Work Audit

**Date**: 2026-03-09

#### Dismissed: Resubmission tests mock resubmitTx instead of network.submitTx
**During**: Test meaningfulness pass — resubmission tests
**Location**: In resubmission tests (`resubmit-each-cycle`, `resubmit-throttle`, `skip-expired`, `revert-resubmit`) at `src/networkClients/UtxoIndex/02g-BlockProcessing.test.ts:~1338-1500`

**Concern**: Tests mock the private `resubmitTx` method directly rather than `network.submitTx`, meaning the CBOR decode → submit path is never exercised. Tests verify resubmission eligibility logic but not the actual submission mechanics.

**Rationale for dismissal**: The test data uses dummy CBOR hex that can't decode. Mocking `resubmitTx` is a pragmatic workaround that lets the tests focus on what they're actually about: eligibility logic (status, throttle, expiry, scope). The submission path coverage gap is already captured in finding gb6mpqbmjv's TODO. The two concerns are the same underlying issue (no valid CBOR in test data), not two separate problems.

#### Dismissed: loadPendingFromStore test doesn't cover provisional entries
**During**: Test meaningfulness pass — startup recovery
**Location**: In `"loadPendingFromStore populates pendingTxHashes on startup"` test at `src/networkClients/UtxoIndex/02g-BlockProcessing.test.ts:~1308`

**Concern**: The test only seeds a pending entry and verifies it's loaded into `pendingTxHashes`. The new code that loads provisional-confirmed entries (`getPendingByStatus("confirmed")` → filter `confirmState === "provisional"` → add to `pendingTxHashes`) is untested.

**Rationale for dismissal**: The new code path is straightforward query-then-filter logic with no branching complexity. The resubmission tests implicitly depend on provisional entries being in `pendingTxHashes` (they seed provisional entries and verify resubmission occurs). Low regression risk.

#### Dismissed: Block-discovered entry deadlineSlot fallback uses block_height as slot
**During**: Requirements vs implementation alignment — REQT/kfemj6eteg
**Location**: In `processTransactionForNewUtxos()` (block-discovered PendingTxEntry creation) at `src/networkClients/UtxoIndex/CachedUtxoIndex.ts:~2307`

**Concern**: The fallback `summary.block_height + this.graceBufferSlots` mixes units — block height is not a slot value. Slots are ~1/sec, heights count blocks.

**Rationale for dismissal**: Fallback only triggers for txs without a validity interval (rare). Block-discovered entries have `status: "confirmed"` — the deadline is used for cleanup timing, not resubmission eligibility. The incorrect value would cause slightly early or late cleanup, not functional failure. And the number of blocks is always going to be less than the number of slots at a given point in time, so the fallback would be conservative (earlier deadline than warranted).

#### Dismissed: confirmState calculation duplicated in three locations
**During**: Cross-file structural quality pass
**Location**: In `updateConfirmationDepths()` (~line 1435), `executeBlockRollback()` re-anchor path (~line 1992), and `processTransactionForNewUtxos()` block-discovered entry (~line 2306) in `src/networkClients/UtxoIndex/CachedUtxoIndex.ts`

**Concern**: The threshold comparison logic (provisional/likely/confident/certain from block depth) appears in three places with minor variations. Could be extracted to a helper.

**Rationale for dismissal**: Only three instances, and each has slightly different context — `updateConfirmationDepths` uses slot-based certainty depth while the others use block-only thresholds. Extracting now would be premature — if a fourth instance appears, it becomes a clear choke-point candidate. Current duplication is within the same file and easily greppable.
