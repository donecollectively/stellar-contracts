## Problem / Context

The `CapoDAppProvider` class exposes its `capo` instance to consumers (via `get capo()`, `useCapoDappProvider` hook, and `onContextChange` тЖТ signals) at intermediate points during its multi-step async initialization sequence. This creates a race condition where consumer components see a capo object that isn't fully initialized тАФ most critically, before wallet attachment completes.

### The Race

1. `connectCapo` creates the capo via `capoClass.createWith(cfg)` and sets it in state
2. `componentDidUpdate` (line 343) fires `onContextChange` when `capo !== prevState.capo`
3. `useCapoDappProvider` hook detects the capo reference change and exposes it to consumers
4. Consumer components (e.g., `MktSaleBuy`) run their effects, see capo exists but `capo.actorContext.wallet` is undefined
5. Consumer sets a **terminal** "no-wallet" state тАФ no mechanism to re-check
6. Wallet attachment completes later, but the consumer has already committed to the wrong state

### Affected Consumers

Every component using the `useCapoDappProvider` тЖТ check capo + wallet pattern:
- `mktSaleBuy.tsx` тАФ sets terminal "no-wallet" phase
- `mktSaleForm.tsx`
- `orgsList.tsx`
- `usePendingChanges.ts`
- `ChangeProposalFormWrapper.tsx`
- `useChangeHistory.ts`
- `MemberStatusPanel.tsx`
- `useFormManager.ts`

Plus signal-based consumers:
- `capoSignals.ts` line 149: `coreSignals.capo.value = provider?.capo`

### Root Cause

The `get capo()` getter (line 2313) returns `this.state.capo` unconditionally тАФ there is no readiness gate. Internal provider code and external consumers share the same accessor, so capo leaks to consumers during setup.

## Plan / Guidance

Gate capo exposure at the provider level using a `rawCapo` / `capo` accessor split with a `capoReady` state flag.

### Step 1: Add `capoReady` to state type and initial state

Add `capoReady?: boolean` to `CapoDappProviderState`. Initialize as `false` (or omitted) in constructor.

### Step 2: Create `private get rawCapo()` accessor

```typescript
private get rawCapo(): CapoType | undefined {
    return this.state.capo;
}
```

### Step 3: Gate `get capo()`

```typescript
get capo(): CapoType | undefined {
    return this.state.capoReady ? this.state.capo : undefined;
}
```

### Step 4: Replace internal `this.capo` references with `this.rawCapo`

Seven references across initialization and wallet setup code:
- Line 950: `if (!this.rawCapo)` (doInitialize)
- Line 1271: `if (this.rawCapo && !this.state.utxoIndex)` (connectWallet)
- Line 1272: `this.rawCapo.setup.network = networkClient` (connectWallet)
- Line 1389: `if (this.rawCapo)` (setWallet)
- Line 1392: `this.rawCapo.setup.network = simpleWallet.cardanoClient` (setWallet)
- Line 1394: `this.rawCapo.setup.actorContext.wallet = wallet` (setWallet)
- Line 1461: `if (this.rawCapo) this.rawCapo.actorContext.wallet = wallet` (setWallet)

Note: `this.state.capo` direct references at lines 1494-1495 (setWallet reconnect check) and line 1524 (`checkWalletTokens`) already bypass the getter and do not need changes.

### Step 5: Set `capoReady: true` in `connectCapo`

After the UTXO caching block completes (~line 1797), both the `!autoNext` path and normal path should include `capoReady: true` in their `updateStatus` state updates alongside `{ capo }`. **Critical**: `capo` and `capoReady` MUST be set in the same state update to ensure `componentDidUpdate` fires `onContextChange` with the capo already visible through the gated getter.

### Step 6: Reset `capoReady: false` on re-entry

At the top of `connectCapo`, reset: include `capoReady: false` in the initial status update (or set it explicitly).

### Step 7: Bootstrap path тАФ no capoReady

When capo is not configured/chartered (early return at ~line 1755), `capoReady` stays `false`. The bootstrap UI operates through class methods that use `rawCapo` internally, not through the public getter.

### Step 8: Verify тАФ no consumer changes

Confirm that the gated getter resolves the race for mktSaleBuy and other consumers without requiring any changes to those files.
