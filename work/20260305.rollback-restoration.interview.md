# Rollback Restoration — Design Interview

## Context
Restoring the suspended rollback functionality in `CachedUtxoIndex.rollbackPendingTx`. The current code only logs diagnostics. The new design must handle contention — where a confirmed on-chain tx spends the same UTxOs our pending tx claimed.

## Established So Far

### Contention Detection Point
In `processTransactionForNewUtxos` (line ~1741), the input-marking loop currently skips UTxOs already marked `spentInTx`. This silently suppresses the signal that a confirmed tx is spending a UTxO our pending tx also claims.

**Resolution**: Change the guard to:
- Unspent UTxO → mark spent by confirmed tx (existing behavior)
- Already marked by same tx → no-op
- Already marked by a *different* tx → **overwrite** spentInTx with confirmed tx hash + record the contention ("pending tx X lost input Y to confirmed tx Z")

The confirmed tx gets the spentInTx marker because it represents on-chain reality. The pending tx's speculative claim yields to evidence.

### Two Regimes
- **No contention**: Pending tx's optimistic updates are fine. Confirmed via `confirmPendingTx` fast path.
- **Contention detected**: A different tx on-chain spent a UTxO our pending tx claimed. Need to hold uncertainty until evidence resolves it (competing tx reaches sufficient block depth).

### Contention Recording
When contention is detected, overwrite the UTxO's `spentInTx` with the confirmed tx hash (on-chain reality wins), and record contention on the losing pending tx's `PendingTxEntry`:

```typescript
contestedByTxs: Array<{ txHash: string, blockHeight: number }>
```

- `blockHeight` is available from the `summary` parameter at detection time
- Parallels existing `confirmedAtBlockHeight` pattern on PendingTxEntry
- At rollback decision time: check `this.lastBlockHeight - entry.blockHeight >= provisionalDepth` (4 blocks) for all entries — all must pass before actual rollback proceeds

### Rollback Decision Flow

**Gate 1 — Expiry depth** (`checkPendingDeadlines`):
Deadline slot must be `provisionalDepth` (default 4) blocks behind `lastBlockHeight`. This ensures we don't give up hope while the deadline is still in shallow blocks. When met → status becomes `rollback-pending`.

**Gate 2 — Contention settlement** (rollback executor, runs each sync cycle):
For `rollback-pending` entries, check if all `contestedByTxs` entries have `this.lastBlockHeight - entry.blockHeight >= provisionalDepth`. If yes (or no contention) → perform actual rollback → `rolled-back`.

Both gates use `this.confirmationThresholds.provisionalDepth` — same configurable threshold, same source of truth.

### State Transitions

**Uncontested failure:**
`pending` → (deadline 4+ blocks deep) → `rollback-pending` → (no contention, next sync) → `rolled-back`

**Contested failure:**
`pending` → (deadline 4+ blocks deep) → `rollback-pending` → (all competing txs at depth ≥ 4) → `rolled-back`

**Success:**
`pending` → `confirmed` → (depth tracking: provisional → likely → confident → certain)

### UI Representation (PendingTxTracker)

Existing symbol map already has `rolled-back → ✗ red (#ef4444)`. Add:
- `rollback-pending` → `✗` grey (#9ca3af) — "we think this failed, waiting for evidence"
- `rolled-back` → `✗` red (#ef4444) — "confirmed failed, rolled back" (existing)

Same symbol, color tells the story. Fits the existing `getStateDisplay` dispatch pattern.

### Actual Rollback Steps (revised from suspended diagnostic code)

**Step 1: Clear spentInTx on uncontested inputs only.**
`findUtxosSpentByTx(failedTxHash)` — by this point, contested inputs already had their spentInTx overwritten to the confirmed tx hash at detection time. So this only finds inputs that weren't contested. Clear them (mark unspent). Safe by construction.

**Step 2: Delete pending-origin UTxOs.**
Remove UTxOs created by the failed tx (utxoId starts with failedTxHash#). These are speculative outputs that never made it on-chain. Safe — confirmed tx outputs have different tx hashes.

**Step 3: Delete pending-origin records.**
Remove records whose utxoId points to the failed tx's outputs. Same rationale as Step 2 — the confirmed tx's records have different utxoIds and are untouched.

**Step 4: Re-parse datums ONLY for restored inputs.**
Only re-parse datums for UTxOs whose spentInTx was cleared in Step 1 (the uncontested ones). Do NOT re-parse inputs whose spentInTx points to a confirmed tx — those were legitimately spent, and their successor records are already correctly indexed by `processTransactionForNewUtxos`.

**Step 5: Set status to "rolled-back".**

**Step 6: Emit `txRolledBack` event.**

**Step 7: In-memory cleanup** (pendingTxMap, pendingSpentUtxoIds).
