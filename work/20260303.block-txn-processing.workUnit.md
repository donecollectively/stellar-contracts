## Problem / Context

### Symptom

A transaction confirmed on-chain remains in the CachedUtxoIndex pending table with status `"pending"`. The `checkPendingDeadlines` timer eventually rolls it back — undoing speculative UTxO state for a transaction that's perfectly fine on-chain.

### Root Cause

`lastBlockHeight` serves dual purpose:
1. **Tip tracking** — `fetchAndStoreLatestBlock()` advances it to the chain tip for `lastSlot`/deadline calculations
2. **Processing cursor** — `checkForNewTxns()` uses `lastBlockHeight + 1` as the `from` parameter for the Blockfrost `addresses/{addr}/transactions` query

These can diverge. `fetchAndStoreLatestBlock()` advances `lastBlockHeight` to the chain tip, but the address/transactions query may not yet have indexed all transactions in that block (Blockfrost indexing lag). On the next sync cycle, `from=lastBlockHeight+1` skips the unindexed transactions. The pending tx is never seen by `processTransactionForNewUtxos`, so `confirmPendingTx` is never called.

### Design Direction

Evolve the data model so that:

1. **Blocks track processing state** — each block in the store has an indicator of whether its transactions have been processed
2. **Block discovery uses `blocks/{hash_or_number}/next`** — fetch the list of blocks we haven't yet seen, based on the latest block we're aware of, rather than relying on address-level transaction queries with a `from` parameter
3. **`lastBlockHeight`/`lastSlot` reflect the chain tip independently** — computed and stored without regard for which blocks have been processed
4. **Processing cursor is separate** — the system knows with high reliability which blocks have actually been processed, and processes each block in order

## Plan / Guidance

### Change 1: Add `processed` indicator to block entries

Add a `processed: boolean` field to `BlockIndexEntry`. When a block's transactions have been fully processed, mark it as processed. Existing blocks in the store should be assumed processed (backward compatibility).

### Change 2: Dual-mode transaction sync

Select sync mode based on gap between processing cursor and chain tip (configurable threshold, default 20 blocks).

#### Incremental mode (gap ≤ threshold)

Block-by-block walk for small gaps (normal operation, 1-2 blocks per poll):

1. **Discover new blocks**: `blocks/{lastProcessedBlockHash}/next` returns up to 100 subsequent blocks.
2. **Check each block for our address**: `blocks/{blockHeight}/addresses` returns `{ address, transactions: [{ tx_hash }] }[]`.
3. **Filter and process**: If `capoAddress` appears, extract tx hashes and process via `processTransactionForNewUtxos()`. Otherwise skip.
4. **Mark processed**: After processing (or skipping), mark the block `processed: true`.

**Wallet address monitoring** (REQT/ygvc43wg0x): The same `blocks/{height}/addresses` response can check registered wallet addresses in the same pass.

#### Catchup mode (gap > threshold)

Address-level bulk sync for large gaps (page reload after offline, startup):

1. **Discover relevant blocks**: `addresses/{capoAddr}/transactions?from={lastProcessedBlock}&order=asc` returns tx summaries with `block_height`. Sparsely populate block records for only the affected blocks.
2. **Process transactions**: Process each discovered transaction via `processTransactionForNewUtxos()`.
3. **Reconcile UTxOs**: Query `addresses/{capoAddr}/utxos` and reconcile against cached UTxO set to ensure correctness (detect spent UTxOs missed during offline period).
4. **Advance cursor**: Mark all blocks through the chain tip as processed.

### Change 3: Decouple tip tracking from processing

`fetchAndStoreLatestBlock()` updates `lastBlockHeight`, `lastBlockId`, and `lastSlot` for tip/time purposes. These values are used for deadline calculations and block-tip polling. They do NOT affect the processing cursor.

`checkForNewTxns()` uses the last **processed** block's hash as its starting point, independent of `lastBlockHeight`.

### Change 4: Simplify periodic refresh to single 5s block poll

Remove the 60s fallback timer entirely — the 5s block-tip poll is the sole trigger for `checkForNewTxns()`. When a new block is detected, it triggers `checkForNewTxns()` which walks forward from the last processed block using `blocks/{hash}/next`.

Add a concurrency guard (`_syncInProgress` flag) so that if `checkForNewTxns` is still running when the next 5s poll fires, the poll skips. This prevents overlapping block walks when processing takes >5s.

### Change 5: Chain rollback detection and recovery

Implement `findIntersection()` to verify chain continuity:

1. **Walk backward from tip**: Query `blocks/{height}/previous` to get recent blocks from the canonical chain.
2. **Compare against stored blocks**: Find the last block where our stored block hash matches the canonical chain (the intersection point).
3. **On mismatch (rollback detected)**:
   - Set `syncState = "recovering from rollback (N blocks)"`
   - Invalidate UTxOs created by transactions in rolled-back blocks (`state=rolled-back`)
   - Clear `spentInTx` markers set by transactions in those blocks (restore previously-spent UTxOs)
   - Invalidate records parsed from UTxOs in those blocks
   - Revert any pending tx confirmations that occurred in those blocks (status back to `pending`)
   - Roll the processing cursor back to the intersection point
   - Emit `chainRollback` event with depth
   - Re-initiate sync from the intersection point

### Change 6: Sync state tracking

Maintain an observable `syncState` property reflecting current sync activity:
- `"idle"` — no sync in progress
- `"syncing"` — incremental block-walk (≤ threshold gap)
- `"catchup sync (N blocks)"` — catchup mode with block count
- `"recovering from rollback (N blocks)"` — rollback recovery with depth

Updated at start and end of each sync activity. Downstream consumers observe this for UI status display.

## Acceptance Criteria

- [ ] A pending tx that lands on-chain is confirmed in the pending table, even if `lastBlockHeight` has already advanced past the block containing it
- [ ] `lastBlockHeight`/`lastSlot` reflect the chain tip independently of block processing state
- [ ] Blocks are processed in order — no block is skipped
- [ ] Each block records which addresses (and tx IDs) were involved
- [ ] The processing cursor (last processed block) is distinct from the tip cursor (`lastBlockHeight`)
- [ ] Processing resumes correctly after restart — no blocks lost between sessions
- [ ] Backward compatibility: existing cached blocks are treated as processed on upgrade
- [ ] When gap > threshold, catchup mode uses address-level query + UTxO reconciliation
- [ ] Chain rollbacks are detected via `findIntersection()` and all affected state is recovered
- [ ] Pending tx confirmations in rolled-back blocks revert to pending
- [ ] `syncState` reflects current sync activity (idle, syncing, catchup, recovering)
- [ ] `chainRollback` event emitted with depth when rollback detected
- [ ] Existing tests pass (02e, 02f) or are updated for the new model
