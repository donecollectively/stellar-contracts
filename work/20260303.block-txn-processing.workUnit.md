## Problem / Context

### Symptom

A transaction confirmed on-chain remains in the CachedUtxoIndex pending table with status `"pending"`. The `checkPendingDeadlines` timer eventually rolls it back ŌĆö undoing speculative UTxO state for a transaction that's perfectly fine on-chain.

### Root Cause

`lastBlockHeight` serves dual purpose:
1. **Tip tracking** ŌĆö `fetchAndStoreLatestBlock()` advances it to the chain tip for `lastSlot`/deadline calculations
2. **Processing cursor** ŌĆö `checkForNewTxns()` uses `lastBlockHeight + 1` as the `from` parameter for the Blockfrost `addresses/{addr}/transactions` query

These can diverge. `fetchAndStoreLatestBlock()` advances `lastBlockHeight` to the chain tip, but the address/transactions query may not yet have indexed all transactions in that block (Blockfrost indexing lag). On the next sync cycle, `from=lastBlockHeight+1` skips the unindexed transactions. The pending tx is never seen by `processTransactionForNewUtxos`, so `confirmPendingTx` is never called.

### Design Direction

Evolve the data model so that:

1. **Blocks track processing state** ŌĆö each block in the store has an indicator of whether its transactions have been processed
2. **Block discovery uses `blocks/{hash_or_number}/next`** ŌĆö fetch the list of blocks we haven't yet seen, based on the latest block we're aware of, rather than relying on address-level transaction queries with a `from` parameter
3. **`lastBlockHeight`/`lastSlot` reflect the chain tip independently** ŌĆö computed and stored without regard for which blocks have been processed
4. **Processing cursor is separate** ŌĆö the system knows with high reliability which blocks have actually been processed, and processes each block in order

## Plan / Guidance

This work unit is phased. Each phase delivers working, testable functionality. Later phases build on infrastructure established by earlier phases.

### Phase 1: Core Block Processing Model

Solves the original bug. Establishes the decoupled cursor model and dual-mode sync.

**REQTs**: fh56sce22g, gfsjgaac1y, 2he55bafxd, 9gq8rwg9ng, 5d4f73c9bf, 9a0nx1gr4b, zzsg63b2fb, jdkjh536mm (sans rollback state)

#### Change 1: Add block status field

Add a `status: "unprocessed" | "processed"` field to `BlockIndexEntry` (not a boolean ŌĆö designed to accommodate `"rolled-back"` in Phase 3 without a schema migration). Existing blocks in the store should be assumed `"processed"` (backward compatibility).

#### Change 2: Dual-mode transaction sync

Select sync mode based on gap between processing cursor and chain tip (configurable threshold, default 20 blocks).

**Incremental mode** (gap Ōēż threshold) ŌĆö block-by-block walk for small gaps (normal operation, 1-2 blocks per poll):

1. **Discover new blocks**: `blocks/{lastProcessedBlockHash}/next` returns up to 100 subsequent blocks.
2. **Check each block for our address**: `blocks/{blockHeight}/addresses` returns `{ address, transactions: [{ tx_hash }] }[]`.
3. **Filter and process**: If `capoAddress` appears, extract tx hashes and process via `processTransactionForNewUtxos()`. Otherwise skip.
4. **Mark processed**: After processing (or skipping), set block `status = "processed"`.

**Wallet address monitoring** (REQT/ygvc43wg0x): The same `blocks/{height}/addresses` response can check registered wallet addresses in the same pass.

**Catchup mode** (gap > threshold) ŌĆö address-level bulk sync for large gaps (page reload after offline, startup):

1. **Discover relevant blocks**: `addresses/{capoAddr}/transactions?from={lastProcessedBlock}&order=asc` returns tx summaries with `block_height`. Sparsely populate block records for only the affected blocks.
2. **Process transactions**: Process each discovered transaction via `processTransactionForNewUtxos()`.
3. **Reconcile UTxOs**: Query `addresses/{capoAddr}/utxos` and reconcile against cached UTxO set to ensure correctness (detect spent UTxOs missed during offline period).
4. **Advance cursor**: Mark all blocks through the chain tip as `"processed"`.

#### Change 3: Decouple tip tracking from processing

`fetchAndStoreLatestBlock()` updates `lastBlockHeight`, `lastBlockId`, and `lastSlot` for tip/time purposes. These values are used for deadline calculations and block-tip polling. They do NOT affect the processing cursor.

`checkForNewTxns()` uses the last processed block's hash as its starting point, independent of `lastBlockHeight`.

#### Change 4: Simplify periodic refresh to single 5s block poll

Remove the 60s fallback timer entirely ŌĆö the 5s block-tip poll is the sole trigger for `checkForNewTxns()`. When a new block is detected, it triggers `checkForNewTxns()` which walks forward from the last processed block.

Add a concurrency guard (`_syncInProgress` flag) so that if `checkForNewTxns` is still running when the next 5s poll fires, the poll skips.

#### Change 5: Sync state tracking

Maintain an observable `syncState` property reflecting current sync activity:
- `"idle"` ŌĆö no sync in progress
- `"syncing"` ŌĆö incremental block-walk (Ōēż threshold gap)
- `"catchup sync (N blocks)"` ŌĆö catchup mode with block count

Updated at start and end of each sync activity. Downstream consumers observe this for UI status display.

#### Change 6: Record confirming block height on pending tx confirmation

When `confirmPendingTx` is called, record `confirmedAtBlockHeight` on the PendingTxEntry. This is forward-compatible infrastructure ŌĆö Phase 2 uses it for depth calculation, Phase 3 uses it to identify confirmations in rolled-back blocks.

#### Phase 1 advisory: delete `highestBlockHeight` dead code

The `highestBlockHeight` variable in the current `checkForNewTxns` is dead code. MUST be deleted during the rewrite, not preserved.

### Phase 2: Confirmation Depth Tracking

Adds graduated confidence for confirmed transactions. Exercises the `confirmedAtBlockHeight` field from Phase 1.

**REQTs**: ddzcp753jr, yn45tvmp6k, thy7tkrxh7, 58b9nzgcbj (amend), fz6z7rr702 (amend), r0y7s2vggr (amend)

#### Change 7: Confirmation depth tracking

Replace binary `"confirmed"` status with graduated confidence based on block depth:

1. When a pending tx is first confirmed on-chain, set `confirmState = "provisional"` and record the confirming block height (already captured in Phase 1).
2. On each sync cycle, recalculate depth for all confirmed-but-not-certain entries as `currentBlockHeight - confirmedAtBlockHeight`.
3. When depth crosses a threshold, advance `confirmState` and emit `confirmStateChanged` event.
4. Thresholds (configurable, defaults): `provisional` < 4, `likely` < 10, `confident` Ōēź 10, `certain` Ōēź 180.
5. App layer may translate these labels to other strings and configure the depth thresholds.

#### Change 8: Amend pending tx events and queries

- `txConfirmed` event includes `confirmState` in payload.
- New `confirmStateChanged` event emitted at depth threshold crossings.
- `getPendingTxs` supports filtering by `confirmState`.

### Phase 3: Chain Rollback Detection & Recovery

Adds robustness for chain reorgs. Builds on block status enum (Phase 1) and depth tracking (Phase 2).

**REQTs**: yasww6cqa4, 4j3rs4pyjt (and all sub-REQTs), dnr06r6ch5, b6h7km6h44, jdkjh536mm (add rollback state)

#### Change 9: Chain rollback detection

Implement `findIntersection()` to verify chain continuity:

1. **Walk backward from tip**: Query `blocks/{height}/previous` to get recent blocks from the canonical chain.
2. **Compare against stored blocks**: Find the last block where our stored block hash matches the canonical chain (the intersection point).

#### Change 10: Rollback recovery

On mismatch (rollback detected):
- Set `syncState = "recovering from rollback (N blocks)"`
- Invalidate UTxOs created by transactions in rolled-back blocks (`state=rolled-back`)
- Clear `spentInTx` markers set by transactions in those blocks (restore previously-spent UTxOs)
- Mark records from UTxOs in those blocks with `state="affected by pending rollback"`
- Revert any pending tx confirmations that occurred in those blocks ŌĆö clear `confirmState`, status back to `pending`
- Mark rolled-back blocks with `status = "rolled-back"` (enum already accommodates this from Phase 1)
- Roll the processing cursor back to the intersection point
- Emit `chainRollback` event with depth
- Re-initiate sync from the intersection point

#### Change 11: Sync state ŌĆö add rollback recovery state

Add `"recovering from rollback (N blocks)"` to the `syncState` values.

## Acceptance Criteria

### Phase 1: Core Block Processing Model
- [ ] A pending tx that lands on-chain is confirmed in the pending table, even if `lastBlockHeight` has already advanced past the block containing it
- [ ] `lastBlockHeight`/`lastSlot` reflect the chain tip independently of block processing state
- [ ] Blocks are processed in order ŌĆö no block is skipped
- [ ] Each block records which addresses (and tx IDs) were involved
- [ ] The processing cursor (last processed block) is distinct from the tip cursor (`lastBlockHeight`)
- [ ] Processing resumes correctly after restart ŌĆö no blocks lost between sessions
- [ ] Backward compatibility: existing cached blocks are treated as processed on upgrade
- [ ] When gap > threshold, catchup mode uses address-level query + UTxO reconciliation
- [ ] `syncState` reflects current sync activity (idle, syncing, catchup)
- [ ] `confirmedAtBlockHeight` recorded when a pending tx is confirmed
- [ ] `highestBlockHeight` dead code removed
- [ ] Existing tests pass (02e, 02f) or are updated for the new model

### Phase 2: Confirmation Depth Tracking
- [ ] Confirmed txs track `confirmState` progressing through provisional ŌåÆ likely ŌåÆ confident ŌåÆ certain
- [ ] `confirmStateChanged` event emitted at each depth threshold crossing
- [ ] Depth thresholds are configurable
- [ ] `txConfirmed` event includes `confirmState`
- [ ] `getPendingTxs` supports filtering by `confirmState`

### Phase 3: Chain Rollback Detection & Recovery
- [ ] Chain rollbacks are detected via `findIntersection()` and all affected state is recovered
- [ ] Pending tx confirmations in rolled-back blocks revert to pending (confirmState cleared)
- [ ] `syncState` gains `"recovering from rollback (N blocks)"` value
- [ ] `chainRollback` event emitted with depth when rollback detected
- [ ] Records from rolled-back blocks marked `state="affected by pending rollback"`
