# Work Unit: Emulator Snapshot Cache Invalidation on Param Changes

**Created**: 2026-03-05
**Status**: Draft
**Area**: stellar-contracts infrastructure (testing/emulator)

## Problem

When a delegate's compile-time const parameters change (e.g., `requiresGovAuthority: true → false` in a `.hlb.ts` bundle class), the emulator's cached bootstrap snapshots (`.stellar/emu/bootstrapWithActors-*`) still contain UTxOs locked by the **old** compiled script hash. The compilation cache (`.stellar/cache/`) can be cleared and new compilations succeed with the updated params, but the emulator snapshot was built against the prior script — so on-chain validation still runs the stale code.

### Observed behavior

1. Changed `requiresGovAuthority = false` in `MarketSale.hlb.ts`
2. Cleared `.stellar/cache/` — scripts recompiled with new params
3. Test still failed: `missing required input with dgTkn capoGov-*` — the on-chain script in the emulator snapshot still enforced `requiresGovAuthority = true`
4. The `Delegate configuration requires upgrade` message appeared in logs, showing the old config vs new config mismatch — but the test proceeded anyway with the old script
5. Even after clearing the emu snapshot (`rm -rf .stellar/emu/bootstrapWithActors-*`), the fresh bootstrap still failed because the upgrade-detection path doesn't handle the mismatch gracefully during test bootstrap

### Root cause

The emulator snapshot cache key doesn't incorporate delegate bundle param hashes. When params change, the cache key still matches, so the stale snapshot is reused. The compiled scripts baked into the snapshot's ref-script UTxOs are from the prior compilation.

## Desired outcomes (investigate one or more)

1. **Self-responsible cache invalidation**: The snapshot cache key should incorporate a hash of all delegate bundle params (or at minimum the full `params` getter output). When any param changes, the cache key changes, forcing a fresh bootstrap. This is the most robust solution — no stale state possible.

2. **Runtime detection and early failure**: When `reusableBootstrap()` loads a cached snapshot, compare each deployed delegate's on-chain config (stored in the charter/delegation datum) against the current bundle's `params`. If they differ, fail immediately with a clear message like `"Emulator snapshot stale: MarketSalePolicy params changed (requiresGovAuthority: true→false). Delete .stellar/emu/ and re-run."` — rather than letting the test proceed to a confusing on-chain validation failure.

3. **Transparent re-bootstrap**: Instead of failing, automatically invalidate the stale snapshot and re-bootstrap when param drift is detected. More user-friendly but harder to implement correctly.

## Context

- Emulator snapshots: `.stellar/emu/bootstrapWithActors-seed42-*`
- Compilation cache: `.stellar/cache/`
- Bundle params flow: `.hlb.ts` → `DelegatedDataBundle.params` getter → `HeliosScriptBundle.paramsToUplc()` → Helios compiler const binding (namespaced as `BasicDelegate::paramName`)
- The `Delegate configuration requires upgrade` log message already detects the mismatch — it just doesn't act on it during test bootstrap

## Additional finding: source code changes (not just params) also stale

Even with `requiresGovAuthority` reverted back to `true` (no param change), after clearing BOTH `.stellar/cache/` AND `.stellar/emu/bootstrapWithActors-*`, and getting a fully fresh bootstrap with new compiled scripts (confirmed by new UTxO hashes in the error), the on-chain script still enforced the old behavior — ignoring new function-based exemption hooks added to `BasicDelegate.hl` and `MarketSalePolicy.hl`.

This means the staleness issue extends beyond just const parameter changes. **Source code changes to `.hl` files** also don't reliably propagate to the on-chain scripts used during test validation, even after clearing all known caches. There may be an additional caching layer (e.g., ref-script UTxOs baked during bootstrap that reference a compiled script hash that doesn't incorporate all transitive source dependencies), or the compilation itself may not be picking up changes to modules imported through the generated `specializedDelegate` wrapper.

## Notes

Discovered during MarketSale gov-authority exemption work. The debugging cost was significant — multiple test cycles with cache clearing before identifying that the emulator snapshot was the stale layer.
