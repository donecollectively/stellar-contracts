## Problem / Context

### Bug 1: Pending txs never confirmed after page reload

After page reload, `loadPendingFromStore()` correctly loads pending tx entries from Dexie and populates `pendingTxHashes`. Then `checkForNewTxns()` runs the block sync. But the block containing the pending tx was already processed in the prior session — it's stored as `state: "processed"` and `getUnprocessedBlocks()` does not return it. The incremental sync only walks unprocessed blocks, so `processTransactionForNewUtxos` is never called for this tx. `confirmPendingTx` is never invoked. The entry stays at `status: "pending"` forever.

This means `updateConfirmationDepths()` (which queries `status === "confirmed"`) never sees the entry. Even when a follow-up tx spending this tx's outputs is already at "likely" depth, the original tx shows "pending" — clearly wrong.

The contract in REQT/fn70x96nxm says: "After first sync cycle: `checkForNewTxns()` MUST confirm any that landed in blocks." This contract is broken because the confirmation mechanism is a side-effect of block processing, not an explicit responsibility.

### Bug 2: Wallet UTxO reconciliation destroys pending tx state

When `syncWalletAddressIfStale()` re-fetches wallet UTxOs from the network, it:
1. Deletes cached UTxOs absent from the fresh network set — including speculative outputs from pending txs
2. Re-indexes fresh UTxOs with `spentInTx: null`, overwriting speculative-spend markers set by `registerPendingTx`

Both behaviors cause the second transaction in a chain to re-spend wallet UTxOs that were consumed by the first (still-pending) tx.

**Partial fixes already applied in this session** (committed with this work unit):
- Wallet reconciliation now preserves `spentInTx` markers when re-indexing fresh UTxOs
- Wallet reconciliation now skips deletion of UTxOs whose txHash is in `pendingTxHashes`
- Capo reconciliation now skips marking UTxOs as spent when their txHash is in `pendingTxHashes`
- Requirements REQT/06b01nyf51, REQT/2he55bafxd, and new REQT/q2aqr5rh7g updated to codify these guards

### Structural problem: `checkForNewTxns` is a misnamed god-method

`checkForNewTxns()` currently orchestrates the entire sync cycle: block discovery, block processing, rollback detection, resubmission, confirmation depth tracking, and rollback settlement. Its name says "check for new transactions" but its scope is far broader. The pending tx confirmation gap exists because confirming pending txs was never an explicit responsibility — it was an accidental side-effect of the block processing path.

### Structural problem: Catchup mode fetches blocks it doesn't need

`fetchAndStoreNewBlocks()` runs unconditionally before mode selection. It walks `blocks/{hash}/next` to enumerate every block in the gap (N/100 API calls), storing each as "unprocessed". Then:
- **Incremental mode** uses these blocks — walks each with `blocks/{height}/addresses` to discover txs. Correct use.
- **Catchup mode** ignores them — uses `addresses/{addr}/transactions` for sparse tx discovery and `addresses/{addr}/utxos` for reconciliation. The only use of the stored blocks is to mark them all "processed" at the end. The block enumeration is wasted API calls and storage in catchup mode.

## Plan / Guidance

### Step 1: Rename and restructure `checkForNewTxns` → `runSyncCycle`

Extract the current monolithic method into a clear orchestration sequence with named steps:

```
runSyncCycle()
├── 1. discoverChainTip()
│      Fetch blocks/latest from Blockfrost.
│      Update IN-MEMORY tip state only: lastBlockHeight, lastBlockSlot,
│      lastBlockId, lastBlockTime.
│      ⚠ MUST NOT store the block in the blocks table — if it did,
│      fetchAndStoreNewBlocks() would walk from the tip and discover
│      zero intermediate blocks, breaking incremental mode.
│      (See Advisory Notes, Finding F1.)
│
├── 2. detectAndHandleRollbacks()
│      Fetch blocks/{tipHash}/previous?count=100, compare against stored blocks.
│      Runs in BOTH modes — effective for gaps up to ~100 blocks (~33 min on Cardano).
│      One API call regardless of gap size.
│
├── 3. syncChainState() — mode selection and execution
│      Determine gap from processing cursor to chain tip.
│      ├── [incremental: gap ≤ catchupThreshold]
│      │     fetchAndStoreNewBlocks()   ← block enumeration ONLY here
│      │     Per-block walk with blocks/{height}/addresses
│      │     processTransactionForNewUtxos() per discovered tx
│      │     Mark each block processed after its txns are handled
│      │
│      └── [catchup: gap > catchupThreshold]
│            addresses/{capoAddr}/transactions?from=N  (sparse tx discovery)
│            processTransactionForNewUtxos() per discovered tx
│            addresses/{capoAddr}/utxos  (reconcile — ground truth for UTxO set)
│            Advance processing cursor to tip (no per-block storage needed)
│
├── 4. tryConfirmPendingTxs()    ← NEW
│      For each entry still at status === "pending" in pendingTxHashes:
│        Query Blockfrost txs/{txHash} to check if it's on-chain.
│        If confirmed: extract block_height from response, call confirmPendingTx(txHash, blockHeight).
│      This is the safety net that catches txs missed by block processing
│      (e.g., after page reload when the confirming block was already processed).
│      In normal operation (no reload), this is a no-op — block processing's
│      fast-path confirm already handles them.
│
├── 5. resubmitStalePendingTxs()
├── 6. updateConfirmationDepths()
└── 7. executeSettledRollbacks()
```

### Step 2: Move block enumeration inside incremental mode only

Currently `fetchAndStoreNewBlocks()` runs before mode selection. Move it inside the incremental branch. Catchup mode does not call it — catchup uses address-level queries and UTxO reconciliation, which don't need per-block records.

### Step 3: Implement `tryConfirmPendingTxs()`

For each txHash in `pendingTxHashes` that still has `status === "pending"` in the store:
1. Query Blockfrost `txs/{txHash}` — returns block info if the tx is on-chain, 404 if not
2. If on-chain: call `confirmPendingTx(txHash, response.block_height)`
3. If 404: skip (tx hasn't landed yet — resubmission and deadline paths handle it)
4. Handle errors gracefully — log and continue to next entry

This step runs AFTER `syncChainState` and BEFORE `updateConfirmationDepths`. In the same cycle, `updateConfirmationDepths` will advance the newly-confirmed entry to the correct `confirmState` based on actual depth.

### Step 4: Update call sites

- **5s periodic timer**: Change from `checkForNewTxns()` to `runSyncCycle()`
- **`syncNow()` startup path**: Change from `checkForNewTxns()` to `runSyncCycle()`
- **Concurrency guard** (`_syncInProgress`): Stays on `runSyncCycle` (same location, renamed)

### Step 5: Update catchup mode cursor advancement

Currently catchup marks all unprocessed blocks as "processed" — but after this change, catchup doesn't store intermediate blocks. The cursor advancement must change:
- Catchup needs to update the processing cursor to the tip block height without having per-block records to mark
- Since `discoverChainTip()` does NOT store the tip block (in-memory only), catchup mode must explicitly create and save the tip block entry as `"processed"` to represent the cursor position. Use the in-memory tip state (`lastBlockHeight`, `lastBlockId`, `lastBlockSlot`, `lastBlockTime`) to construct the `BlockIndexEntry`.

### Important constraints

- **Rollback detection MUST run in both modes.** It is 1 API call (`blocks/{tipHash}/previous?count=100`) that produces real results for gaps up to ~100 blocks. Do not skip it in catchup mode. It uses the in-memory `lastBlockId` set by `discoverChainTip()` — no block store dependency.
- **`processTransactionForNewUtxos` fast-path confirm stays.** The existing code that confirms pending txs during block processing is correct and efficient for the normal (non-reload) case. `tryConfirmPendingTxs` is the safety net, not a replacement.
- **`tryConfirmPendingTxs` must be efficient.** In normal operation (no reload), there should be few or zero entries at `status === "pending"` after block processing. The Blockfrost call only happens for genuinely unconfirmed entries.
- **The 10s timer for `checkPendingDeadlines()` is independent** and unchanged. It is NOT part of `runSyncCycle`.

### Pre-Work Collaborative Review (2026-03-10)

#### Architect leads — findings

##### Finding F1: `discoverChainTip()` must not store to the block table

The plan's restructured flow puts the chain tip discovery as Step 1, before mode selection and block processing. The current `fetchAndStoreLatestBlock()` does two things: updates in-memory tip state AND saves the block to the store. The new `discoverChainTip()` must split these apart.

**Why**: `fetchAndStoreNewBlocks()` walks forward from the **highest stored block** via `store.getLatestBlock()`. If `discoverChainTip()` stores the tip block, `fetchAndStoreNewBlocks()` starts from the tip and discovers zero intermediate blocks — breaking incremental mode.

**Resolution**: `discoverChainTip()` updates in-memory state only (`lastBlockHeight`, `lastBlockSlot`, `lastBlockId`, `lastBlockTime`). No write to the blocks table. Cascading implications addressed in plan Steps 1 and 5:
- Incremental mode: `fetchAndStoreNewBlocks()` still walks from the last stored block (previous tip), discovers intermediates naturally, and the new tip is discovered during the walk.
- Catchup mode: must explicitly construct and save the tip block as `"processed"` from in-memory state to advance the cursor.
- Rollback detection: unaffected — uses in-memory `lastBlockId` for `blocks/{tipHash}/previous?count=100`.

> **Resolution**: Plan updated — `updateChainTip` renamed to `discoverChainTip` with explicit no-store constraint documented in Step 1 and Step 5.

### Coder Advisory — Key Implementation Pitfalls

**1. `discoverChainTip()` is in-memory only — do NOT write to the blocks table.**
The current `fetchAndStoreLatestBlock()` does both (updates in-memory state AND saves to store). The new `discoverChainTip()` must only do the first half: fetch `blocks/latest`, update `lastBlockHeight`/`lastBlockSlot`/`lastBlockId`/`lastBlockTime`. If you store the block, `fetchAndStoreNewBlocks()` starts its walk from the tip and discovers zero intermediate blocks — incremental mode breaks silently.

**2. `fetchAndStoreNewBlocks()` invariant: walks from `store.getLatestBlock()`.**
This method walks forward from the highest stored block. After `discoverChainTip()` (which doesn't store), the highest stored block is still the previous tip. `fetchAndStoreNewBlocks()` discovers intermediates naturally and the new tip during its walk. Do not change this method's starting-point logic.

**3. Catchup mode must explicitly store the tip block as `"processed"`.**
Since `discoverChainTip()` doesn't store anything, and catchup mode skips `fetchAndStoreNewBlocks()`, the tip block has no store entry. Catchup mode must construct a `BlockIndexEntry` from the in-memory tip state and save it as `"processed"` to advance the cursor. Without this, the next sync cycle sees no processed blocks and falls back into catchup mode again.

**4. `fetchAndStoreLatestBlock()` still exists — don't remove it.**
It's called during `syncNow()` (full initial sync). Only the periodic `runSyncCycle()` uses the new `discoverChainTip()`. The two methods serve different lifecycle stages.

**5. Ordering within `runSyncCycle()` matters.**
`tryConfirmPendingTxs()` must run AFTER `syncChainState()` (so block-processing fast-path confirm runs first) and BEFORE `updateConfirmationDepths()` (so newly confirmed entries get depth tracking in the same cycle). The plan's step ordering is intentional — don't rearrange.
