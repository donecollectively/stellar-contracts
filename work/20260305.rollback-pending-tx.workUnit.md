## Problem / Context

### Suspended rollback functionality

`CachedUtxoIndex.rollbackPendingTx` currently only logs diagnostics — the actual rollback steps (clearing spentInTx, deleting speculative outputs/records, re-parsing datums, emitting events) are all commented out with "SUSPENDED" markers. Real-world experience has built confidence that this functionality should be restored.

### Silent contention suppression

In `processTransactionForNewUtxos`, the input-marking loop at line ~1741 checks `if (existingUtxo && !existingUtxo.spentInTx)` — it skips UTxOs already marked spent. When a pending tx speculatively claimed a UTxO, and a *different* confirmed tx on-chain actually spends it, the confirmed tx's spend is silently ignored. The UTxO stays marked as spent by the pending tx (the wrong one), making a naive rollback dangerous — it would incorrectly restore a UTxO that was genuinely consumed on-chain.

### Need for evidence-based rollback

Two sources of uncertainty must be resolved before rollback:
1. **Expiry depth**: A deadline slot passing in a shallow block might still be reversed. We should wait until the deadline is `provisionalDepth` (4) blocks deep before giving up hope.
2. **Contention settlement**: When a competing confirmed tx took our inputs, we should wait until that tx's block is also `provisionalDepth` deep before proceeding, to be sure the on-chain reality is settled.

## Plan / Guidance

### Design Decisions (from design interview)

Full design rationale is captured in `.proj/work/20260305.rollback-restoration.interview.md`.

### Change 1: Contention detection in processTransactionForNewUtxos

In the input-marking loop (~line 1741), change the guard from skipping already-marked UTxOs to:
- Unspent → mark spent by confirmed tx (existing behavior)
- Already marked by same tx → no-op
- Already marked by a *different* tx → overwrite spentInTx with confirmed tx hash, look up the pending tx that previously claimed it (via old spentInTx value), and append `{ txHash: confirmedTxHash, blockHeight }` to that PendingTxEntry's `contestedByTxs`.

### Change 2: PendingTxEntry type update

Add `"rollback-pending"` to the status union. Add `contestedByTxs?: Array<{ txHash: string, blockHeight: number }>` field (optional, default empty).

### Change 3: Depth-based expiry in checkPendingDeadlines

Replace the current deadline comparison with a depth-based check: deadline slot must be at least `confirmationThresholds.provisionalDepth` blocks behind `lastBlockHeight`. When met, set status to `"rollback-pending"` instead of calling `rollbackPendingTx` directly.

### Change 4: Rollback settlement executor (new method)

New method, called each sync cycle alongside `updateConfirmationDepths`. For each `rollback-pending` entry:
- If `contestedByTxs` is non-empty, check that all competing txs have `lastBlockHeight - blockHeight >= confirmationThresholds.provisionalDepth`
- If any are too shallow, skip (re-check next cycle)
- When clear: execute the actual rollback (Changes 5-8)

### Change 5: Safe spentInTx clearing

Un-suspend Step 1. `findUtxosSpentByTx(txHash)` only finds UTxOs still marked as spent by the failed tx (contested ones were already overwritten at detection time). Clear those — this is safe by construction.

### Change 6: Pending output/record cleanup

Un-suspend Steps 2-3. Delete UTxOs and records where utxoId starts with `txHash#`. Confirmed tx artifacts have different tx hashes and are unaffected.

### Change 7: Selective datum re-parse

Un-suspend Step 4 with restriction: only re-parse datums for input UTxOs whose spentInTx was cleared in Change 5. Do NOT re-parse inputs whose spentInTx points to a confirmed tx — their successor records are already correctly indexed.

### Change 8: Finalization

Un-suspend Steps 5-7. Set status to `"rolled-back"`, emit `txRolledBack` event, clean up in-memory state.

### Change 9: PendingTxTracker UI

Add `"rollback-pending": { symbol: "✗", color: "#9ca3af", label: "Rollback Pending" }` to the `stateDisplay` map. Add dispatch case in `getStateDisplay`.

## Advisory Notes

### Design Interview (2026-03-05)

Full design exploration captured in `.proj/work/20260305.rollback-restoration.interview.md`. Key design decisions:

1. **Contention detection point**: `processTransactionForNewUtxos` input-marking loop — confirmed tx overwrites spentInTx, records contention on losing pending entry
2. **Two-gate rollback**: Gate 1 (expiry depth ≥ provisionalDepth) → `rollback-pending` → Gate 2 (all competing txs at depth ≥ provisionalDepth) → actual rollback
3. **Safe rollback by construction**: contested inputs already have spentInTx overwritten at detection time; `findUtxosSpentByTx` only returns uncontested inputs; datum re-parse restricted to restored inputs only
4. **Uniform path**: all expired txs go through `rollback-pending` (uncontested ones pass through quickly on next sync cycle)
5. **Both gates use same threshold**: `confirmationThresholds.provisionalDepth` (default 4 blocks)

**Test Sketch — Contention-Aware Two-Gate Rollback**

The existing `02f-PendingTx.test.ts` has register/confirm/rollback/deadline/startup tests. New tests extend this file with two new describe groups.

```
describe("contention detection (REQT/hhbcnvd9aj)")
  
  "confirmed tx overwrites pending spentInTx and records contestedByTxs"
    - Register pending tx (marks inputs spent by pendingTxHash)
    - Simulate confirmed tx spending same input via processTransactionForNewUtxos
    - Assert: input's spentInTx overwritten to confirmedTxHash
    - Assert: pending entry's contestedByTxs contains { confirmedTxHash, blockHeight }
    - Assert: pending entry status still "pending" (contention recorded, not rolled back yet)

  "confirmed tx for same txHash is a no-op (fast-path confirm, no contention)"
    - Register pending tx
    - Process same txHash through processTransactionForNewUtxos
    - Assert: fast-path confirm fires, no contestedByTxs, status "confirmed"

  "multiple confirmed txs contesting different inputs accumulate in contestedByTxs"
    - Register pending tx with 2+ inputs
    - Simulate two different confirmed txs each taking one input
    - Assert: contestedByTxs has 2 entries with distinct txHashes and blockHeights

describe("two-gate rollback lifecycle (REQT/a9y19g0pmr)")

  "uncontested expiry: deadline at depth → rollback-pending → rolled-back next cycle"
    - Register pending tx, no contention
    - Advance chain past deadline by provisionalDepth blocks
    - checkPendingDeadlines → assert status "rollback-pending"
    - Run rollback executor → assert status "rolled-back"
    - Assert: inputs restored, outputs deleted, event fired

  "deadline not deep enough stays pending"
    - Register pending tx
    - Advance chain past deadline but only by 1 block (< provisionalDepth)
    - checkPendingDeadlines → assert status still "pending"

  "contested expiry: Gate 2 holds until competing tx at depth"
    - Register pending tx, simulate contention at blockHeight X
    - Advance chain past deadline by provisionalDepth
    - checkPendingDeadlines → "rollback-pending"
    - Run rollback executor with lastBlockHeight only 2 blocks past X → assert still "rollback-pending"
    - Advance to X + provisionalDepth → run executor → assert "rolled-back"
    - Assert: contested input NOT restored (spentInTx still confirmedTxHash)
    - Assert: uncontested inputs restored

  "selective datum re-parse: only uncontested inputs re-parsed"
    - Register pending tx with Capo attached, 2 inputs with inline datums
    - Simulate contention on input A, leave input B uncontested
    - Run full rollback
    - Assert: record for input B re-parsed (exists in records table)
    - Assert: record for input A NOT re-parsed (confirmed tx's record intact)
```
