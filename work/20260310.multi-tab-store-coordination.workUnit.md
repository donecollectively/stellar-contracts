## Problem / Context

### Problem 1: Redundant & Contentious Sync

Multiple browser tabs sharing the same IndexedDB all independently run `runSyncCycle()`, making redundant Blockfrost API calls and potentially interleaving writes. This wastes API quota and creates write contention.

### Problem 2: Non-Atomic Multi-Step Writes

`registerPendingTx()` performs multiple sequential Dexie writes (save pending entry, mark inputs spent, index outputs, parse records, save tx CBOR) without any atomicity guarantee. A page reload mid-write leaves the store in an inconsistent state — the `PendingTxEntry` may exist without all its output UTxOs indexed. The `confirmPendingTx` fast-path then assumes outputs are present and skips re-indexing (now mitigated by REQT/05lkfji14c output verification, but prevention is better than repair).

### Design

**Two-layer coordination:**

1. **Sync mutex** (Phase 1): A metadata-table row (`syncMutex`) establishes which browser tab is responsible for sync activity. The owning tab freshens the mutex each sync cycle. Non-owning tabs serve reads from cache and poll the mutex for staleness (15s threshold). On `beforeunload`, the owner removes the row (best-effort). Stale or missing mutex triggers acquisition attempt from any tab.

2. **Write lock** (Phase 2): A uniform `withWriteLock(pid, activityName, tables, callback)` gate for all multi-step writes. Any tab — sync owner or not — acquires the write lock before performing multi-step writes. The lock uses a metadata-table row (`writeLock`) with pid, activityName, and timestamp. Stale after 1s. For Dexie, the callback additionally runs inside a `db.transaction('rw', tables, callback)` for atomicity. For non-transactional stores, the metadata-row protocol alone provides mutual exclusion.

**All writers go through the write lock** — the sync mutex only establishes sync responsibility, it does not grant implicit write access.

#### Code Whisperer — sequenced follow-up

##### Finding F1: Mutex CAS requires Dexie transaction wrapping

The Plan/Guidance describes compare-and-swap semantics for mutex acquisition ("read the current mutex, verify it is absent or stale, then write"). REQT/f3w3hkjt4t specifies this correctly. However, the Plan does not explicitly note that the CAS read-check-write sequence must execute inside a Dexie `db.transaction('rw', ['metadata'], ...)` to prevent a TOCTOU race where two tabs both read a stale/absent mutex and both write. Without transactional wrapping, the CAS is cosmetic — both tabs could acquire simultaneously. The Coder needs this signpost.

> **tessa**: This also affects testability — concurrent acquisition attempts resulting in exactly one winner is a natural test case.

### Suggestions & Potential Audit Concerns

#### Suggestions
- (low) `markInitialSyncDone` and `waitForInitialSyncDone` access the metadata table via cast to `any`. Consider adding generic `getMetadata`/`setMetadata` methods to `UtxoStoreGeneric`.
- (low) The `withWriteLock` spin loop has no timeout — a permanently held lock (impossible with 1s stale break, but defensive) would spin forever. Consider adding a max wait timeout.

#### Concerns
- (medium) Dexie ambient transactions: when `withWriteLock` wraps a callback in `db.transaction('rw', tables, ...)`, all Dexie operations from `CachedUtxoIndex` that call store methods will join that transaction. If any store method touches a table not listed in the transaction's scope, Dexie will throw. The auto-inclusion of `metadata` and `logs` covers the known cases, but future store methods touching other tables inside a locked callback would need the tables list updated.
