## Problem / Context

### Observed Problem

A transaction that is successfully submitted to the Cardano network occasionally vanishes — it never appears on-chain. This is consistent with a short-lived micro-fork: the node that accepted the transaction was on a fork that lost the chain race. The transaction left the mempool (because it was "included" in a block) but the block was orphaned.

### Current Behavior

The existing system has no detection or recovery for this scenario:
- **TxSubmitMgr** confirms the tx via `getTx()` — which may reflect the losing fork's state
- **CachedUtxoIndex** never sees the tx in a canonical block during its block-walk
- Eventually the tx's deadline expires and the existing rollback lifecycle (REQT/a9y19g0pmr) rolls it back — but by then, the validity window has typically closed and the tx is irrecoverable

### Desired Behavior

Two coordinated behaviors address this:

1. **Quiet resubmission**: CachedUtxoIndex proactively resubmits any pending transaction (including those with shallow "provisional" confirmation) back to the network on each sync cycle. If the tx is already confirmed on the winning chain, the resubmit harmlessly fails. If the confirming block was orphaned, the resubmit gets the tx back into the mempool for inclusion in a future block. This runs independently of rollback detection — it's a standing resilience behavior that can get a tx into the next block faster than waiting for rollback detection.

2. **Micro-fork rollback detection**: CachedUtxoIndex detects when stored blocks are no longer on the canonical chain by comparing against Blockfrost's `blocks/{tip}/previous?count=100`. For rolled-back blocks, it resolves each affected tx's confirmation uniformly (self-submitted and block-discovered alike): if the tx appears in a canonical replacement block, it re-anchors the confirmation to the new block; if not, it reverts to `"pending"` status — making the tx eligible for resubmission. Rollback detection never deletes UTxOs, records, or spent markers directly; that cleanup only happens through the existing pending tx rollback lifecycle (REQT-1.11.3) when a tx genuinely fails to land.

### Key Design Decision: `pendingTxHashes` Until "Likely"

Currently, `confirmPendingTx()` removes the txHash from `pendingTxHashes` immediately upon first confirmation. This means a provisionally-confirmed tx drops out of all "pending" scope — including resubmission eligibility — at the exact moment when it's most vulnerable to micro-fork orphaning.

The fix: keep txs in `pendingTxHashes` until `confirmState` reaches `"likely"` (depth ≥ `provisionalDepth`, default 4 blocks). This means:
- `isPending()` returns truthy for provisionally-confirmed txs — correct, because they're not yet reliably confirmed
- The resubmission loop naturally covers them without special-case queries
- `updateConfirmationDepths()` performs the `pendingTxHashes.delete` when advancing to "likely"

### Future Compatibility

This design anticipates:
- **Ogmios as additional submitter**: `resubmitTx()` method abstracts the submission path; today it's `this.network.submitTx()`, tomorrow it routes through multiple submitters
- **Dred tx relay**: externally-received txs enter through `registerPendingTx` (or a lighter variant) and land in the same `pendingTxHashes` + Dexie table, automatically eligible for resubmission
- **Full chain rollback detection (REQT-1.12)**: this work unit implements the micro-fork-relevant subset; deeper rollback recovery can extend the same detection mechanism

### Known Limitation

A tx confirmed in an orphaned block at depth ≥ `provisionalDepth` (reached "likely" before the rollback) would have been removed from `pendingTxHashes` and would NOT be resubmitted by the quiet resubmission loop. Full recovery requires the rollback detection path to revert the confirmation (REQT/2grpnzb2q0). In practice, Cardano micro-forks resolve within 1-3 blocks, well within the provisional window. Deeper rollbacks are extraordinarily rare.

## Plan / Guidance

### Phase 1: PendingTxEntry Fields & pendingTxHashes Lifecycle

1. Add optional fields to `PendingTxEntry`: `confirmedInBlockHash?: string`, `forkRecoveryCount?: number`, `lastResubmitAt?: number`
2. Modify `confirmPendingTx()`:
   - Store `confirmedInBlockHash` from the block being processed
   - **Remove** the `pendingTxHashes.delete(txHash)` call
   - Keep `pendingTxMap.delete` and `cleanupPendingSpentUtxoIds` as-is (session-scoped, not needed for resubmission)
3. Modify `updateConfirmationDepths()`:
   - When advancing `confirmState` to `"likely"`: call `pendingTxHashes.delete(txHash)`
4. Verify `isPending()` behavior: it now returns truthy for provisionally-confirmed txs — confirm this is correct for all callers

### Phase 2: Quiet Resubmission

1. Add `private async resubmitTx(signedCborHex: string): Promise<void>` — decodes CBOR, calls `this.network.submitTx()`, swallows expected errors (UTxO-already-spent, expiry)
2. Add `private async resubmitStalePendingTxs(): Promise<void>`:
   - For each txHash in `pendingTxHashes`:
     - Load `PendingTxEntry` from store
     - Skip if `lastResubmitAt` was < 10s ago
     - Skip if validity window expired (`lastValidSlot < lastBlockSlot`)
     - Skip if deadline block has reached "likely" depth (same check as rollback trigger)
     - Call `resubmitTx(entry.signedTxCborHex)`
     - Update `entry.lastResubmitAt = Date.now()`, save to store
3. Call `resubmitStalePendingTxs()` from `checkForNewTxns()` — after block processing, before `updateConfirmationDepths()`
4. Also call from `resolvePendingState()` for page-reload recovery

### Phase 3: Rollback Detection

1. Add `private async detectRolledBackBlocks(): Promise<string[]>` (returns rolled-back block hashes):
   - Fetch `blocks/{lastBlockId}/previous?count=100`
   - Sort response by `height` ascending
   - For each canonical block: compare hash against `store.findBlockByHeight(height)`
   - If stored block exists AND hash differs → collect the stored block's hash as rolled-back
2. Call `detectRolledBackBlocks()` from `checkForNewTxns()` — before block processing (so we clean up before indexing new blocks)

### Phase 4: Rollback Execution

1. Add `private async executeBlockRollback(rolledBackBlockHashes: string[], canonicalBlocks: BlockDetailsType[]): Promise<void>`:
   - Fetch `blocks/{canonicalBlockHash}/txs/cbor` for each canonical replacement block → build a map of `txHash → { blockHash, blockHeight, blockSlot, cbor }`
   - Fetch `blocks/{rolledBackBlockHash}/txs` for each rolled-back block → collect all txHashes affected
   - For each affected txHash:
     - Load `PendingTxEntry` from store (all txs should have one — block processing creates them)
     - If txHash found in canonical tx map → **re-anchor**: update `confirmedInBlockHash`, `confirmedAtBlockHeight`, `confirmedAtSlot`, recalculate depth/`confirmState`
     - If txHash NOT in canonical tx map → **revert**: call `revertConfirmedPendingTx(entry)`
   - Mark each rolled-back block as `state: "rolled back"` via `store.saveBlock()`
   - Save each canonical replacement block as `state: "unprocessed"` for normal sync to reprocess
   - Emit `chainRollback` event
2. Add `private async revertConfirmedPendingTx(entry: PendingTxEntry): Promise<void>`:
   - Set `status = "pending"`, clear `confirmedAtBlockHeight`, `confirmedAtSlot`, `confirmedInBlockHash`, `confirmState`, reset `confirmationBlockDepth = 0`
   - Increment `forkRecoveryCount`
   - Save to store
   - Re-add to `pendingTxHashes`
   - Do NOT touch UTxOs or records — speculative state remains correct

**Note — block-discovered PendingTxEntries**: `processTransactionForNewUtxos()` MUST create a `PendingTxEntry` for every tx it processes (with `status: "confirmed"`, `confirmState` based on depth). CBOR is available from Blockfrost via `blocks/{hash}/txs/cbor` or `txs/{hash}/cbor`. This ensures all txs — self-submitted and block-discovered — have entries for uniform rollback resolution.

### Ordering in checkForNewTxns()

```
checkForNewTxns():
  1. fetchAndStoreNewBlocks()          // discover new blocks
  2. detectRolledBackBlocks()          // compare canonical vs stored  
     → executeBlockRollback() for each  // clean up rolled-back blocks
  3. syncIncremental/syncCatchup       // process unprocessed blocks (including canonical replacements)
  4. resubmitStalePendingTxs()         // resubmit pending + provisional txs
  5. updateConfirmationDepths()        // advance confirmState, remove from pendingTxHashes at "likely"
  6. executeSettledRollbacks()          // existing deadline-based rollback
```

## Advisory Notes

### Design Discussion (2026-03-09)

Collaborative design session between Architect, Code Whisperer, and Coder exploring the problem space and converging on this approach. Key decisions:

1. **CachedUtxoIndex owns resubmission** — it has the signed CBOR, block-depth knowledge, and sync loop. TxBatcher/TxSubmitMgr handle initial submission; CachedUtxoIndex handles persistent lifecycle.

2. **Resubmission is independent of rollback detection** — it runs as a standing resilience behavior every sync cycle. This gets txs back into the mempool faster than waiting for rollback detection to trigger.

3. **Resubmit provisional txs too** — txs with `confirmState === "provisional"` are in the micro-fork danger zone. Keeping them in `pendingTxHashes` until "likely" depth naturally includes them in resubmission scope.

4. **`blocks/{tip}/previous?count=100` for detection** — single API call covers any realistic rollback depth. `blocks/{hash}/txs` for discovering which txs were in a rolled-back block.

5. **Uniform tx treatment** — all txs (self-submitted, block-discovered, future Dred-relayed) get `PendingTxEntry` records. Rollback resolution is uniform: re-anchor if found on winning fork, revert to pending if not. Rollback detection NEVER touches UTxOs/records/spent-markers — that cleanup only happens through the existing pending tx rollback lifecycle (REQT-1.11.3).

6. **Re-anchor as common path** — most micro-fork rollbacks simply move the tx from one block to another. Re-anchoring patches confirmation metadata without disrupting the user. Only txs genuinely lost from the chain get the revert-to-pending treatment.

7. **CBOR always available** — Blockfrost provides CBOR via `blocks/{hash}/txs/cbor` (batch) and `txs/{hash}/cbor` (individual), so all `PendingTxEntry` records can have `signedTxCborHex` regardless of origin.

8. **Future-compatible** — `resubmitTx()` abstracts submission path for future Ogmios/Dred multi-path routing. Dred-relayed txs enter the same pending lifecycle.
