## Reviewer Verification

### Architect — zero findings

Verified architectural alignment across all Phase 1 changes:

- **Decoupled cursor model**: Clean separation confirmed — `lastBlockHeight`/`lastBlockId`/`lastSlot` track chain tip independently, `getLastProcessedBlock()` provides processing cursor via `[state+height]` compound index. No interference between the two.
- **ARCH-fk1fgq7map boundary respected**: CachedUtxoIndex remains the sole component with Helios coupling. New block processing methods work through existing `fetchFromBlockfrost` and `processTransactionForNewUtxos` pathways. No new Helios coupling introduced.
- **Dual-mode sync architecture**: Mode selection based on unprocessed block count (store-based) rather than raw height gap — a healthy refinement from the plan that's more precise. Both modes correctly converge on the same state (all blocks processed, cursor at tip).
- **Block discovery separation**: `fetchAndStoreNewBlocks()` cleanly separates discovery (walking `blocks/{hash}/next`) from processing (`syncIncremental`/`syncCatchup`). This was a refinement during implementation, not an explicit pre-work plan item, but architecturally sound and well-motivated.
- **Forward-compatible design**: `BlockState` enum (`"unprocessed" | "processed" | "rolled back"`) and `confirmedAtBlockHeight`/`confirmState` on PendingTxEntry correctly scaffold Phase 2 and Phase 3 without requiring future schema migrations.
- **No unvetted architectural decisions**: All changes trace to the pre-work plan or are well-motivated refinements within scope.

### Code Whisperer — zero findings

Structural quality review of the changeset:

- **Code structure**: `checkForNewTxns()` has clean control flow — concurrency guard → discovery → mode selection → cleanup in `try/finally`. No scattered heuristics; block processing logic centralized in `syncIncremental()`/`syncCatchup()`, discovery centralized in `fetchAndStoreNewBlocks()`.
- **Error handling**: `fetchAndStoreNewBlocks()` failure gracefully falls back to `fetchAndStoreLatestBlock()` for tip-only tracking. System retries on next poll cycle. Robust.
- **Naming stewardship**: `syncIncremental`, `syncCatchup`, `fetchAndStoreNewBlocks`, `reconcileUtxos` form a coherent, domain-descriptive vocabulary. No library overloading.
- **Schema migration**: v4 migration replaces `hash, height` with `hash, [state+height]` compound index. Verified no remaining code queries blocks by height alone — old index usage fully migrated.
- **`fetchAndStoreLatestBlock()` race prevention**: Now preserves existing block state when the block is already in store, preventing overwrites of "processed" state by tip fetch.
- **Testability seams**: `fetchFromBlockfrost` and `processTransactionForNewUtxos` are clean mock points. New methods are testable via existing spy patterns.
- **Dead code**: `highestBlockHeight` confirmed removed with test verification.
- **Implementation evolution from pre-work plan**: Block discovery as a separate step (`fetchAndStoreNewBlocks`) was a refinement, not a divergence — no structural concerns introduced.
