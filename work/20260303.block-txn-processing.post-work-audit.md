## Post-Work Finding: Deadline/Tip Race (Phases 1+2)

**Date**: 2026-03-04
**Identified by**: User observation (spurious rollbacks), Work Planner code review
**Result**: Fixed

### Finding: F-PW1 — checkPendingDeadlines uses tip slot instead of last processed slot

**Status**: resolved
**Location**: In `checkPendingDeadlines()` (pending tx deadline expiry check) at `src/networkClients/UtxoIndex/CachedUtxoIndex.ts`

**Observation**: `checkPendingDeadlines()` compared `entry.deadline` against `this.lastSlot` — the chain tip slot. But `this.lastSlot` advances via `fetchAndStoreNewBlocks()` as soon as new blocks are discovered, *before* those blocks are processed by `syncIncremental()`/`syncCatchup()`. This creates a race: the deadline timer (every 40s) fires, sees `deadline < tipSlot`, and rolls back the transaction — even though the confirming block is sitting in the store as "unprocessed" and hasn't been walked yet. This is the exact same tip-vs-cursor divergence the work unit was created to fix, manifesting in the deadline path.

**Resolution**: Changed `checkPendingDeadlines()` to query `getLastProcessedBlock()` and use its slot for deadline comparison instead of `this.lastSlot`. Updated `rollbackPendingTx()` log messages to reference the processed slot. Updated 02f test to save processed blocks in the store alongside `setLastSyncedBlock()` calls. Also changed deadline check interval from 10s to 40s — with the 5s block-tip poll driving sync, 10s was unnecessarily aggressive and gave less time for block processing to confirm pending txns before the deadline check fires.
