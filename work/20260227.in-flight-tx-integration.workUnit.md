## Problem / Context

The CachedUtxoIndex serves as the Capo's network client (`ReadonlyCardanoClient`) but has no awareness of locally-submitted transactions. After a user submits a transaction, the index is blind to it for up to 60 seconds (next Blockfrost poll). This causes:

1. **Stale reads after writes** — queries return old state, confusing users and risking duplicate submissions
2. **Double-spend risk** — transaction builders may select UTXOs already consumed by a pending tx
3. **No pending-state visibility** — no way for UI to show "pending" indicators or defer conflicting operations

The solution connects the transaction submission path to the UTXO index: submitted transactions are decoded via existing pipelines, their inputs marked as speculatively spent and outputs indexed as pending. Each transaction has a deadline (based on chain time, not wallclock) after which unconfirmed entries are rolled back. Events notify the CapoDappProvider for ✓ confirmation toasts and recovery UI on rollback.

### Key Design Decisions (from architecture session)

- **CBOR as integration format** — submission side passes signed CBOR hex; no Helios coupling leak
- **No new fields on UtxoIndexEntry/RecordIndexEntry** — pending state is a property of the transaction (PendingTxEntry table), not the UTXO
- **Reuse `spentInTx`** for speculative spends — the pending-tx table determines if a spend is speculative vs confirmed
- **`isPending()` on CachedUtxoIndex**, not on `FoundDatumUtxo` — keeps FoundDatumUtxo as a plain type
- **Chain-time deadlines** — `txValidityEnd + graceBuffer` compared to last synced block slot time
- **10s deadline timer** — separate from 60s Blockfrost poll; also runs 72h purge
- **Register after successful submit** — avoids rollback path for submission errors
- **In-memory `Map<txHash, txd>`** for live TxDescription; session-scoped, not persisted
- **CapoDappProvider mediates** all UI access to index events

## Plan / Guidance

### Implementation Strategy: Bottom-Up by Layer

Work proceeds bottom-up through 4 phases. Each phase produces testable, committable work. Phase boundaries are chosen so each layer is complete and independently verifiable before the next layer builds on it.

### Phase 1: Type & Store Foundation

**Goal**: PendingTxEntry type exists, store interface is extended, Dexie implementation is complete. No CachedUtxoIndex changes yet.

1. Create `types/PendingTxEntry.ts` with the type definition (Helios-free, IndexedDB-serializable)
2. Extend `UtxoStoreGeneric` interface with pending-tx methods:
   - CRUD: `savePendingTx`, `findPendingTx`, `getPendingByStatus`, `setPendingTxStatus`
   - Rollback: `clearSpentByTx`, `deleteUtxosByTxHash`, `deleteRecordsByTxHash`
   - Cleanup: `purgeOldPendingTxs`
3. Implement all methods in `DexieUtxoStore`:
   - Add `pendingTxs` table with `txHash` PK and `status` index
   - Add `spentInTx` index to utxos table (enables efficient `clearSpentByTx` rollback queries; `null` values excluded from IndexedDB indexes so cost is minimal)
   - Schema version bump (v3)
   - Implement all interface methods

**Verification**: Store methods can be unit-tested in isolation with a test database.

### Phase 2: Core CachedUtxoIndex Integration

**Goal**: Transactions can be registered, confirmed, and rolled back. The index correctly reflects pending state in its stored data.

1. Add in-memory `pendingTxMap: Map<string, TxDescription>`
2. Implement `registerPendingTx()`:
   - Decode CBOR via `decodeTx()`
   - Compute deadline (`txValidityEnd + graceBuffer` from decoded tx validity interval)
   - Persist PendingTxEntry to store (status: "pending")
   - Mark inputs: set `spentInTx` on each consumed UTXO
   - Index outputs: run through `indexUtxoFromOutput()`
   - Parse records if Capo attached
   - Store `txd` in pendingTxMap
3. Modify `checkForNewTxns()`: when a txHash matches a pending entry, skip re-indexing and call `setPendingTxStatus("confirmed")`
4. Implement `checkPendingDeadlines()` on 10s timer:
   - Query pending entries where deadline < last synced block slot time
   - For each: `clearSpentByTx`, `deleteUtxosByTxHash`, `deleteRecordsByTxHash`, `setPendingTxStatus("rolled-back")`
   - Timer lifecycle: start and stop alongside the 60s refresh timer in `startPeriodicRefresh()`/`stopPeriodicRefresh()` — one start call manages both, one stop call clears both. Use `.unref?.()` on the 10s timer for Node.js test environments.

**Verification**: Register a pending tx, verify UTXOs/records appear in store with correct spentInTx state. Simulate confirmation by feeding the same txHash through checkForNewTxns. Simulate rollback by advancing past deadline.

### Phase 3: Query & State API

**Goal**: Consumers can check pending state and the index exposes its sync status.

1. Implement `isPending(item: TxOutputId | string | FoundDatumUtxo)` — synchronous, uses pendingTxMap
2. Implement `getPendingTxs()` — returns pending entries from store
3. Implement `pendingSyncState` getter — tracks stale/fresh lifecycle
4. Wire purge into the 10s timer — `purgeOldPendingTxs(72h)`

**Verification**: isPending returns correct results for pending-origin UTXOs, speculatively-spent UTXOs, and confirmed UTXOs.

### Phase 4: Events & Startup Recovery

**Goal**: Downstream consumers receive notifications. Page-reload recovery works.

1. Extend EventEmitter type with `txConfirmed`, `txRolledBack`, `pendingSynced`
2. Wire event emission into confirmation path (Phase 2 confirm) and rollback path (Phase 2 rollback)
3. Implement startup recovery:
   - On construction, load pending entries from Dexie
   - Set `pendingSyncState` to "stale"
   - After first sync cycle, resolve pending state and flip to "fresh"
   - Emit `pendingSynced`

**Verification**: Simulate page reload by constructing a new CachedUtxoIndex against a Dexie DB that already has pending entries. Verify events fire and state transitions correctly.

## Implementer's Notes

### IN-1: Record restoration gap in rollback plan

**Discovered during**: Phase 2 implementation, with Code Whisperer structural review

**Problem**: The architecture doc's rollback dataflow (steps 1-3) and REQT-1.11.3/a9y19g0pmr do not account for restoring **previous versions of records** that were overwritten during `registerPendingTx`.

**Root cause**: Records use `id` as PK in Dexie. When a pending tx spends a UTXO containing record "foo" and creates a new output with an updated "foo", `saveRecord()` uses `put()` which **overwrites** the original record. On rollback, `deleteRecordsByTxHash(txHash)` only deletes records with `utxoId` starting with `txHash#` (pending-origin records). If the new record overwrote the old one (same `id`, different `utxoId`), deleting the new record leaves no record at all — the original data is lost.

By contrast, UTXOs don't have this problem because inputs and outputs have different `utxoId`s — the old UTXO is restored by clearing `spentInTx`, the new one is deleted. No data loss.

**Affected artifacts**:
- `utxoIndex.ARCHITECTURE.md` — "Dataflow: Rollback Expired Pending Transaction" section needs a step 4 for record re-parsing
- `utxoIndex.arch.jsonl` — if the rollback interaction is captured there
- REQT-1.11.3/a9y19g0pmr — rollback steps should include record re-parsing

**Resolution applied in implementation**: After deleting pending-origin records and restoring speculatively-spent UTXOs, the rollback procedure re-parses inline datums from restored input UTXOs (if Capo is attached) to recreate the overwritten records. The re-parsing naturally produces `spentInTx: null` since `parseAndSaveRecord` always creates fresh records. The adjusted rollback sequence is:

1. `clearSpentByTx(txHash)` — restore speculatively-spent UTXOs
2. `deleteUtxosByTxHash(txHash)` — remove pending-origin UTXOs
3. `deleteRecordsByTxHash(txHash)` — remove pending-origin records
4. **Re-parse datums from restored input UTXOs** — decode the rolled-back tx's CBOR to find its inputs, look up the now-restored UTXOs, and re-parse their inline datums via `parseAndSaveRecord()` (requires Capo attached; no-op without Capo, which is consistent since records wouldn't have been parsed during registration either)
5. `setPendingTxStatus(txHash, "rolled-back")`
6. Emit `txRolledBack`

**Architecture doc update needed**: Someone should update the architecture doc's rollback dataflow and the Pending State Model section's "Pending records" paragraph to reflect step 4. The current text only says "On rollback, records whose `utxoId` starts with the rolled-back txHash are deleted" — it should also specify re-parsing of restored input UTXOs' datums.

### IN-2: `lastSlot` not advancing during periodic refresh

**Discovered during**: Phase 2 implementation, Code Whisperer structural review

**Problem**: `lastSlot` was only set during `syncNow()` (constructor) and `fetchAndStoreLatestBlock()` (full sync path). The 60s periodic refresh calls `checkForNewTxns()` which never updated `lastBlockHeight` or `lastSlot`. This means during ongoing operation, `lastSlot` was stale — it never advanced beyond the value from initial sync.

**Impact on in-flight feature**: `checkPendingDeadlines()` compares `entry.deadline < this.lastSlot`. With a stale `lastSlot`, deadlines would never trigger even after the chain advanced past them.

**Pre-existing**: This gap existed before the in-flight feature, but was harmless because nothing depended on `lastSlot` advancing during periodic refresh. The `now` property (which returns `lastSlot`) would also return stale values.

**Resolution applied**: At the end of `checkForNewTxns()`, if new transactions were processed, fetch block details for the highest `block_height` seen in the transaction summaries. This updates `lastBlockHeight`, `lastBlockId`, and `lastSlot`. Costs one additional Blockfrost API call per sync cycle that discovers new transactions (none when idle).

**Architecture doc update**: Not strictly needed — this is an implementation detail of `checkForNewTxns` correctness. But the `now` property documentation could note that it reflects the latest synced block, advancing during both initial sync and periodic refresh.

**Discovered during**: Test development — the existing `uutMintingMintDelegate.hl` and `DelegatedDatumTester.hl` did not support `CreatingDelegatedData` activity, preventing creation of datum-bearing records in tests.

The Capo's `requiresDelegateAuthorizingMint()` expected the mint delegate to use a lifecycle activity, but `CreatingDelegatedData` is not a lifecycle activity — so validation failed with "unexpected delegate activity".

3. Bumped `builderVersion` on `snapToInstallingTestDataPolicy` and `snapToInstalledTestDataPolicy` in `CapoForDgDataPolicyTestHelper.ts` to invalidate cached snapshots built with old scripts.

**Impact**: These Helios changes affect the test infrastructure only — `uutMintingMintDelegate.hl` and `DelegatedDatumTester.hl` are both in `src/testing/`. No production on-chain code was modified. The changes align the test delegates with the pattern used in real downstream dApps (S3).
