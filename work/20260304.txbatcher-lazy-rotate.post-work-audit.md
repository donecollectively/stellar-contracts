## Findings

### Finding: F1 — `cancel()` drops `"rotated"` event — TxBatchUI subscriber not notified
**Status**: resolved

**Location**: In `TxBatcher.cancel()` (manages batch cancellation) at `src/networkClients/TxBatcher.ts:114-122`

**Observation**: The pre-work analysis stated "no external subscribers exist currently" for the `"rotated"` event. This was incorrect. `src/ui/TxBatchUI.tsx:182` subscribes to `txBatcher.$notifier.on("rotated", ...)` and uses it to call `setTxBatch(batch)` — updating React state so the UI reflects the current batch. The refactored `cancel()` nulls `_current` without emitting `"rotated"`, leaving TxBatchUI's React state pointing to the old destroyed BSC. The `rotate()` path is unaffected — `StellarTxnContext`'s constructor immediately accesses `this.currentBatch` after rotate, triggering the getter and the event. But `cancel()` is a terminal UI action with no downstream getter access.

**Resolution**: Widened `TxBatcherChanges.rotated` type to `[BatchSubmitController | undefined]`. `cancel()` now emits `"rotated"` with `undefined`. `TxBatchUI` listener guards `batch` before accessing `$allTxns`/`$txChanges`; `setTxBatch(undefined)` clears React state, making `hasBatch` false and hiding the panel.
