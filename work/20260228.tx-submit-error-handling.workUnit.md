## Problem / Context

When a transaction is submitted to Blockfrost through the UI and Blockfrost rejects it, the transaction can be resubmitted repeatedly even though it cannot possibly succeed. The user sees no clear indication of why this happens from the UI — the tx just keeps cycling.

Investigation traced three interacting defects in the tx submission state machine (`TxSubmitMgr`):

### Finding 1: `doSubmit()` silently swallows errors

`TxSubmitMgr.doSubmit()` has a `try/catch` where the outer `catch` logs diagnostics (error message, CBOR hex dump) but **never rethrows**. If `submitTx()` throws synchronously (rather than returning a rejected promise), `doSubmit()` resolves to `undefined`. This means:

- `notSubmitted()` is never called (the `.catch` handler in `trySubmit()` doesn't fire)
- No state transition occurs — the state machine gets stuck in `"submitting"` permanently
- No retry, no failure — silent dead end

**Fix**: Add `throw e;` after the diagnostic logging in the catch block. This is already implemented on the working branch.

### Finding 2: `"failed"` state has an unconditional infinite reconfirm loop

`onEntry.failed` always schedules a deferred `"reconfirm"` transition after 10 seconds, and the `failed` state's transition table permits `reconfirm → confirming`. This creates a cycle:

```
failed → (10s) → reconfirm → confirming → tryConfirm() →
  notConfirmed → notOk → (isBadTx?) → failed → (10s) → ...
```

Even when `isBadTx` is set (the tx is known to be permanently invalid), this loop runs forever. The tx is not resubmitted in this path (that's guarded), but it cycles through confirmation attempts indefinitely, generating network traffic and UI state churn.

The loop cannot be made unconditionally terminal because `isBadTx` classification can be wrong — Blockfrost's `UtxoFailure` response doesn't distinguish "already submitted through another path" from "input UTxO already spent." A brief rescue window is needed.

**Fix**: Replace the unconditional reconfirm with a two-branch dispatch:
1. **`isBadTx` set** → bounded rescue: dedicated `rescueAttempts` counter with firm backoff (1.27²), max ~5 attempts (~2.5 min window), then terminal `failed`
2. **else** → existing reconfirm behavior (legitimate recovery path)

Note: an earlier design included a `!successfulSubmitAt` → direct resubmit branch, but analysis showed it's unreachable — the `confirming.notOk` handler already redirects never-submitted txs back to `submitting` before they reach the `failed` state. A `"resubmit"` transition is wired in the table for future use but nothing currently triggers it.

### Finding 3: Blockfrost errors outside UTxO/validity fall through to plain `Error`

`BlockfrostV0Client.submitTx()` matches on `OutsideValidityIntervalUTxO` and `UtxoFailure` to throw typed errors (`SubmissionExpiryError`, `SubmissionUtxoError`). All other 400-level responses throw a plain `Error` with no `kind` field.

In `TxSubmitMgr.notSubmitted()`, a plain `Error` fails both `isExpiryError()` and `isUnknownUtxoError()` checks and falls through to:
```ts
this.$mgrState.isBadTx = problem;
this.transition("failed");
```

This means any Blockfrost rejection that isn't UTxO or validity related (e.g., `ValueNotConservedUTxO`, budget exceeded, etc.) immediately enters the infinite reconfirm loop from Finding 2.

**Fix**: This is addressed by Finding 2's bounded rescue — once the failed state is no longer an infinite loop, unrecognized errors will cycle briefly and then terminate. No changes needed in `notSubmitted()` itself; the error classification is correct (it IS a bad tx), the problem is downstream in the failed state's behavior.

### Combined effect

For a tx rejected by Blockfrost with a non-UTxO, non-validity error:
1. Blockfrost returns 400 → throws plain `Error`
2. `notSubmitted` doesn't recognize the `kind` → sets `isBadTx`, transitions to `"failed"`
3. `"failed"` onEntry schedules reconfirm in 10s
4. Reconfirm → confirming → can't find tx on-chain → notOk sees `isBadTx` → back to `"failed"`
5. Repeat forever

## Plan / Guidance

Two surgical changes to `src/networkClients/TxSubmitMgr.ts`:

### Change 1: Fix silent error swallowing in `doSubmit()`

Add `throw e;` at the end of the outer `catch` block in `doSubmit()`. All existing diagnostics (logging, hex dump, debugger) are preserved — the error simply stops being swallowed after them.

**Already implemented** on the working branch.

### Change 2: Bounded rescue + direct resubmit in `failed` state

Four sub-changes, all in `TxSubmitMgr.ts`:

1. **Add `rescueAttempts: 0` to `SubmitManagerState` type and `resetState()`** — dedicated counter that isn't inflated by confirmation timeouts (unlike `confirmationFailures` which increments multiple times per rescue cycle due to the 3s `pendingActivity` timeout)

2. **Add `"resubmit"` transition** — add to `TxSubmitterTransitions` union type, to `noTransitionsExcept`, and to the `failed` state's transition table as `resubmit: { to: "submitting" }`

3. **Replace `onEntry.failed`** with two-branch conditional:
   - `isBadTx` → increment `rescueAttempts`; if under budget (~5), schedule deferred `"reconfirm"` with firm backoff; if over budget, do nothing (terminal)
   - else → existing `"reconfirm"` behavior unchanged

4. **Verify `submitting` onEntry** handles re-entry correctly — it calls `trySubmit()` which should work on re-entry since it creates a new `pendingActivity`

## Acceptance Criteria

- [ ] `doSubmit()` catch block rethrows after diagnostics — synchronous `submitTx` failures propagate to `notSubmitted()`
- [ ] A tx marked `isBadTx` that enters `failed` state terminates after a bounded rescue window (~5 attempts, ~2.5 minutes)
- [ ] A tx that fails for legitimate recoverable reasons (slot battle, transient service error) retains existing retry behavior
- [ ] A misclassified "bad tx" (actually submitted through another path) can still be discovered during the rescue window via confirmation check
- [ ] `rescueAttempts` counter is not inflated by confirmation timeouts within a single rescue cycle

### Suggestions & Potential Audit Concerns

#### Suggestions

- **(low)** `gradualBackoff()` at line 623 accepts an optional `backoff` parameter but always logs as `gradualBackoff(...)` even when called with `firmBackoff`. The log output shows `gradualBackoff(140 * 1.6129 ^ 1 = 140)` which is technically correct but could be confusing when reading logs — the label says "gradual" but the exponent is firm. Consider parameterizing the log label.

- **(medium)** The `else` branch in `onEntry.failed` (non-`isBadTx` path) retains the original unbounded reconfirm. This is correct for the current use cases (slot battles, transient issues where the tx was successfully submitted), but could theoretically loop forever if a tx reaches `failed` via `expirationDetected` without `isBadTx` set — a narrow edge case reachable through `confirming.notOk` when `expirationDetected` is true. In practice this is harmless (the backoff caps at `maximum` and the tx will eventually expire from the validity window), but a future hardening pass could add a ceiling here too.

- **(low)** The `"resubmit"` transition is wired but unreachable. This is intentional defensive code for future error classification improvements, but it's worth a comment in the transition table noting that it's not currently triggered — already added in the `failed` state's table entry.

#### Concerns

- **(medium)** The `confirmationFailures` counter increments inside `confirming.notOk` on every failed confirmation, including the rapid-fire ones during the 3s `pendingActivity` timeout. Over 5 rescue cycles, this counter can reach 10+ even though only 5 "real" rescue attempts occurred. Code that reads `confirmationFailures` (e.g., the `confirmationFailures > 10` check in `confirming.notOk.onTransition` line ~903) may interact unexpectedly with the rescue window — if `confirmationFailures` exceeds 10 during rescue, the path redirects to `submitting` instead of `failed`, bypassing the rescue budget. This didn't surface in testing (the fast test intervals don't accumulate enough timeout-driven increments), but could matter with production timing. The `rescueAttempts` counter was introduced specifically to avoid this coupling, but the `confirmationFailures > 10` redirect is an independent path that doesn't check `isBadTx`.
