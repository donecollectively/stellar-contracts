## Problem / Context

When `Capo.mkTxnUpgradeIfNeeded()` runs auto-setup for delegated data controllers, each controller's `setupCapoPolicy()` generates up to three transactions via `mkTxnInstallPolicyDelegate`:

1. **Queue delegate change** — mints UUT, updates charter with pending change (`mkTxnQueuingDelegateChange`)
2. **Ref script creation** — stores the new delegate's compiled script on-chain at the Capo address (`txnMkAddlRefScriptTxn`)
3. **Commit pending changes** — activates the queued change in the manifest (`commitPendingChangesIfNeeded`)

These are registered as deferred `addlTxn`s and executed in sequence. If any transaction fails to submit (wallet funds insufficient, network error, user cancels), the remaining transactions in the chain may or may not execute depending on the failure mode. Two specific failure scenarios are not recovered:

### Problem 1: Orphaned pending changes never get committed

If transaction #3 (commit) fails to submit but #1 (queue) succeeded, the charter has a pending change that was never activated. On next startup:

- `setupCapoPolicy` calls `mkTxnInstallPolicyDelegate` → `mkTxnQueuingDelegateChange`
- `mkTxnQueuingDelegateChange` detects the existing pending change at line 4463 and throws `AlreadyPendingError`
- `AlreadyPendingError extends TxNotNeededError` → silently skipped
- The orphaned pending change is never committed
- The `commitPendingChangesIfNeeded` at line 2879 (after the autoSetup loop) WOULD catch this — it re-reads `findCharterData()` and checks `pendingChanges.length > 0` — **but only if the delegate's addlTxn was skipped cleanly**

**Subtlety**: The post-loop `commitPendingChangesIfNeeded` at line 2879 does handle this case for orphaned commits. However, it runs AFTER the autoSetup loop, meaning any delegate whose upgrade depends on the orphaned change being committed first (e.g., a second delegate that needs the first one active) would fail. Moving a commit check BEFORE the loop ensures orphaned changes are cleared before any new setup runs.

### Problem 2: Missing ref script is never backfilled

If transaction #2 (ref script) fails to submit but #1 and #3 succeed, the delegate is active in the manifest but has no ref script on-chain. On next startup:

- `setupCapoPolicy` calls `mkTxnInstallPolicyDelegate` → `mkTxnQueuingDelegateChange`
- The delegate exists and its script hash matches → `TxNotNeededError("Policy doesn't need an update")` at line 4343 or 4376
- This skips the **entire** `mkTxnQueuingDelegateChange` function — including `txnMkAddlRefScriptTxn` at line 4487
- No ref script is created
- The delegate permanently operates without a ref script

**Runtime impact**: `txnAttachScriptOrRefScript` (line 3207) has a fallback — when the ref script is missing, it adds the script bytes directly to the transaction with a `⚠️ missing refScript` warning. This works but bloats every transaction that uses the delegate, increasing fees. The upgrade detection path (line 1929) throws `unexpected: refScript for ${role} not found` which is harder — it prevents detecting future upgrades for that delegate.

### Combined scenario

Both problems can occur together: queue succeeds, ref script fails, commit fails. On next run, the orphaned pending change blocks the upgrade path, and no ref script exists. The system is stuck.

## Plan / Guidance

### Change 1: Pre-loop orphaned pending changes commit

In `mkTxnUpgradeIfNeeded()`, add a `commitPendingChangesIfNeeded` call BEFORE the autoSetup loop. This clears any orphaned pending changes from a previous failed run before new delegate setup begins.

**Implementation detail**: `includeAddlTxn` enforces unique names. Parameterize `commitPendingChangesIfNeeded` with an optional `name` argument (default: `"commit pending charter changes"` to preserve existing behavior). The pre-loop call passes `"commit orphaned pending changes"` as the name. Both calls share the same underlying `mkTxnCommittingPendingChanges()` logic.

If there are no orphaned changes, the pre-loop commit throws `TxNotNeededError` and is cleanly skipped. The post-loop commit remains as-is to handle freshly-queued changes from the autoSetup loop.

### Change 2: Ref script backfill in setupCapoPolicy

In `DelegatedDataContract.setupCapoPolicy()`, modify the `mkTcx` callback to catch `TxNotNeededError` from `mkTxnInstallPolicyDelegate` and check whether the delegate's ref script exists on-chain. Three outcomes:

1. **Upgrade needed** → existing path runs (queue + ref script + commit)
2. **No upgrade needed, ref script exists** → `TxNotNeededError` (skip entirely)
3. **No upgrade needed, ref script missing** → return just the ref script creation txn

**Structural sketch** — the try/catch lives inside the `mkTcx` callback because the addlTxn pipeline (`resolveMultipleTxns`) expects `mkTcx` to either return a tcx or throw `TxNotNeededError`:

```typescript
mkTcx: async () => {
    try {
        const charterData = await this.capo.findCharterData();
        return await this.capo.mkTxnInstallPolicyDelegate({...});
    } catch (e) {
        if (!(e instanceof TxNotNeededError)) throw e;
        // backfill check: does ref script exist?
        const script = await this.asyncCompiledScript();
        const hash = script.hash();
        const capoUtxos = await this.capo.findCapoUtxos();
        const refUtxo = await this.capo.findRefScriptUtxo(hash, capoUtxos);
        if (refUtxo) throw e; // ref script exists — skip cleanly
        return this.capo.mkRefScriptTxn(script); // backfill
    }
},
```

### Not in scope

- Creating a Capo architecture JSONL or reqts file — that's a larger effort
- Handling ref script backfill for core delegates (minter, capo, mintDelegate) — those are created during charter creation, not autoSetup
- Retry/recovery for the queue transaction itself (#1) — if the UUT mint fails, there's nothing to backfill

## Acceptance Criteria

- [ ] Orphaned pending changes from a previous failed run are committed before the autoSetup loop processes any delegates
- [ ] If no orphaned pending changes exist, the pre-loop commit is cleanly skipped via `TxNotNeededError`
- [ ] A delegate that is current (no upgrade needed) but missing its ref script gets the ref script created via a single backfill transaction
- [ ] A delegate that is current AND has its ref script produces a clean `TxNotNeededError` skip (no unnecessary transactions)
- [ ] The full upgrade path (queue + ref script + commit) continues to work as before when an upgrade IS needed
- [ ] `AlreadyPendingError` for a delegate with orphaned pending changes no longer silently skips — the pre-loop commit resolves it before the delegate's setup runs

#### Code Whisperer — sequenced follow-up

##### Finding F1: Backfill detection should use `this`, not getDgDataController

**Severity**: Concern

Change 2's plan (steps 2-4) routes through `capo.getDgDataController(recordTypeName, { onchain: true })` to obtain the delegate's compiled script for ref script checking. But `setupCapoPolicy()` already runs on the delegate instance — `this` IS the delegate. The method has direct access to its own compiled script via `this.getBundle()` and `this.asyncCompiledScript()`.

The plan then warns (in the Note) that `onchain: true` triggers upgrade detection which throws `unexpected: refScript for ${role} not found` — the very error the backfill is trying to prevent. The plan suggests `onchain: false` as an alternative but doesn't resolve the contradiction.

Using `this` directly eliminates the onchain:true/false dilemma entirely, avoids coupling to the upgrade detection path, and satisfies REQT/75md751tp4 (Upgrade Detection Safety) cleanly.

> **Resolution**: Plan steps 2-3 replaced with `await this.asyncCompiledScript()` — eliminated `getDgDataController` indirection and the `onchain: true/false` Note entirely. Step 3 also updated to fetch fresh `capoUtxos` (addresses F2 preemptively). Commit: see below.

##### Finding F2: Fresh capoUtxos needed inside deferred mkTcx callback

**Severity**: Concern

The backfill detection needs `capoUtxos` for `findRefScriptUtxo(hash, capoUtxos)`. The `setupCapoPolicy` method receives `capoUtxos` in its options, but the mkTcx callback runs deferred during `resolveMultipleTxns` — potentially after other delegates have created ref script UTxOs that change the on-chain UTxO set. The plan doesn't address data freshness — the backfill check should fetch fresh `capoUtxos` via `this.capo.findCapoUtxos()` inside the mkTcx callback.

> **Resolution**: Addressed in F1's plan update — step 3 now reads `capo.findRefScriptUtxo(hash, await capo.findCapoUtxos())`, fetching fresh UTxOs inside the deferred callback.

##### Finding F3: Clarify commitPendingChangesIfNeeded adaptation

**Severity**: Suggestion

`commitPendingChangesIfNeeded` hardcodes the addlTxn name `"commit pending charter changes"`. The pre-loop call needs a different name per REQT/0crhes9x8a. Two clean approaches: (a) parameterize the method with an optional name argument (default preserves current behavior), or (b) make the pre-loop call inline via `tcx.includeAddlTxn()` with the underlying `mkTxnCommittingPendingChanges()`. Option (a) is DRYer.

> **Resolution**: Plan updated to specify option (a) — parameterize with optional name argument, default preserves existing behavior.

##### Finding F5: Backfill error handling structure needs explicit placement guidance

**Severity**: Concern

The backfill detection logic (catching `TxNotNeededError` from `mkTxnInstallPolicyDelegate`, checking ref script, either rethrowing or returning a backfill tcx) must live inside the `mkTcx` callback — not at the `setupCapoPolicy` level. This is because the addlTxn resolution pipeline (`resolveMultipleTxns`) calls `mkTcx` and expects either a returned tcx (submit this txn) or a thrown `TxNotNeededError` (skip this txn). The plan describes the three outcomes but doesn't specify the structural placement:

- The try/catch wraps the `mkTxnInstallPolicyDelegate` call **inside** the existing `mkTcx` callback
- When ref script exists: the rethrown `TxNotNeededError` propagates up through `mkTcx` to `resolveMultipleTxns`, which handles it as a clean skip — this is not recursive, it's the addlTxn pipeline's normal "not needed" path
- When ref script is missing: the `mkTcx` callback returns the backfill tcx (from `capo.mkRefScriptTxn`), which the pipeline submits as the delegate's only transaction

Without this placement guidance, the Coder might put the try/catch at the wrong level and break the addlTxn pipeline's error contract.

> **Resolution**: Added structural sketch to the plan showing the try/catch inside the mkTcx callback, with the TxNotNeededError propagation path annotated.
