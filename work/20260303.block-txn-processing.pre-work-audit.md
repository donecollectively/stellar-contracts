#### Finding: F1 â€” No concurrency guard on checkForNewTxns; redundant 60s fallback
**Status**: resolved
**Location**: In `startPeriodicRefresh()` (manages polling timers) at `src/networkClients/UtxoIndex/CachedUtxoIndex.ts:553`

**Observation**: The 5s block-tip poll and 60s fallback timer both call `checkForNewTxns()` independently with no concurrency guard. With the new block-walk model, concurrent walks could process the same blocks twice. The 60s fallback is redundant with the 5s block-tip poll.

> **Cwispy**: The `setInterval` pattern fires regardless of async callback completion â€” even a single timer needs a guard if processing can exceed the interval.

**Resolution**: User directed: remove the 60s fallback entirely. Add `_syncInProgress` concurrency guard. Updated work unit plan (Change 4), REQT/zzsg63b2fb (simplified to block-tip-only with guard), added REQT/zzsg63b2fb to work unit scope. #### Finding: F2 â€” API call budget during long catchup
**Status**: resolved
**Location**: Plan / Guidance, Change 2 (block-by-block processing)

**Observation**: (Concern) After long offline periods, block-by-block walk means 1000+ `blocks/{height}/addresses` calls. This applies even after a short offline period like a page reload.

> **Archie**: The block-walk is only high-value for incremental updates (1-2 blocks). For any meaningful gap, address-level queries are more efficient.
