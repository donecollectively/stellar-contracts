## Findings

#### Finding: F1 — No concurrency guard on checkForNewTxns; redundant 60s fallback
**Status**: resolved
**Location**: In `startPeriodicRefresh()` (manages polling timers) at `src/networkClients/UtxoIndex/CachedUtxoIndex.ts:553`

**Observation**: The 5s block-tip poll and 60s fallback timer both call `checkForNewTxns()` independently with no concurrency guard. With the new block-walk model, concurrent walks could process the same blocks twice. The 60s fallback is redundant with the 5s block-tip poll.

> **Cwispy**: The `setInterval` pattern fires regardless of async callback completion — even a single timer needs a guard if processing can exceed the interval.

**Resolution**: User directed: remove the 60s fallback entirely. Add `_syncInProgress` concurrency guard. Updated work unit plan (Change 4), REQT/zzsg63b2fb (simplified to block-tip-only with guard), added REQT/zzsg63b2fb to work unit scope.
